# AA-155 — S0 Dedup Detection + Review Flow

## Decisions

- `normalize_group_key()` extracted as a standalone function in `handler.py` (not a class method) so unit tests can import it directly without mocking asyncpg.
- Branching happens in `process_file()` (handler layer), not inside `RawTourRepository`. The repository stays a dumb INSERT helper; the staging logic belongs to the ingestion orchestrator.
- Old case-sensitive dedup in `RawTourRepository.insert()` (lines 32-41) is removed — it's now superseded by the process_file() branching + the `uq_raw_tours_active_per_group` unique index from migration 052.
- `source_group_id`, `source_version`, `source_status` columns added to INSERT in `repo.insert()` — they arrive pre-set from `process_file()` via the data dict.
- Dry-run dedup check (in `admin_pipeline.py:ingest_s3`) also normalized so preview classification matches actual commit behavior.
- Decision endpoints placed in `admin_pipeline.py` (not `v1_s0.py`) — they are admin-only (X-Admin-Secret auth), not tenant JWT.
- `DELETE /admin/raw-tours/{tour_id}` checks FK on generated_content (active published row) before trashing — 409 if blocked.
- Duplicate Review in frontend is a new Step 5 (not a modal) to keep the existing 4-step flow intact. Step 5 only appears when `tours_staged > 0`.

## Changed

- `RawTourRepository.insert()`: removed old dedup SELECT, added source_group_id/version/status columns.
- `handler.py:process_file()`: split batch into new vs staged; returns `tours_written`, `tours_staged`, `staged_ids`.
- `admin_pipeline.py:ingest_s3()`: normalized dry-run dedup; enriched non-dry-run response with `tours_staged` fields.
- `admin_pipeline.py`: 4 new endpoints added (upload-staging GET/POST, raw-tours set-active/delete).
- `upload/page.tsx`: Step 5 Duplicate Review added after commit if `tours_staged > 0`.

## Tradeoffs

- Staging in DB vs in-memory: DB staging means decisions persist across browser refreshes. Cost: one extra table query per duplicate on upload. Worth it — upload batches are small (< 500 tours).
- `uq_raw_tours_active_per_group` partial unique index (migration 052) enforces invariant at DB level. `set-active` endpoint still does explicit UPDATE before setting active row to avoid constraint violation sequence.
- "keep_both" decision: both rows get source_status='superseded'. Active is determined by sku (non-null) > ingest_at DESC. This is consistent with the migration 052 backfill logic.

## Should know

- Migration 052 (`upload_staging` table + lifecycle columns on raw_tours) must be applied to dev DB before testing. It was merged in AA-154.
- `process_file()` is called both from `ingest_s3` endpoint and from the Lambda handler — Lambda handler ignores the new return fields (backward compatible).
- dry_run=True path in `ingest_s3` still shows duplicates as "blocked" in preview (reason: "duplicate_tour"). After this ticket, these are tours that would be STAGED not rejected. Frontend badge still shows "Duplicate" which is accurate.
