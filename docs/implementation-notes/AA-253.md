# AA-253 — BFF Proxy Route Auth (X-Admin-Secret gated by requireAdmin)

## Decisions

- **New shared helper `frontend/lib/auth-server.ts`** (`requireAdmin`, `requireTenant`) instead of
  inlining the verify-call in each route — 4-5 routes needed the identical pattern (read cookie →
  POST `/auth/verify-admin` → fail closed on missing/invalid/network-error). `requireTenant` created
  for symmetry per the task spec but not wired into any route yet — `tenant/[...path]`'s tenant
  branch was already correct (forwards `cis_tenant_token` as Bearer, backend verifies) and the task
  explicitly said not to touch it.
- **`admin/[...path]/route.ts`: dropped the `x-admin-username` header.** The old code decoded
  (unverified) JWT payload for both `sub` → `x-admin-user-id` and `username` → `x-admin-username`.
  `requireAdmin()`'s return type only carries `adminId`/`role` (per task spec — `VerifyAdminResponse`
  has `username` too, but the header was never read anywhere — grepped `x-admin-username` across
  `api/` and `frontend/`, zero hits). Rather than mislabel `role` into that header slot or extend
  the return type for a value nothing consumes, the header was removed.
- **`tenant/pipeline/brand-identity/route.ts` gated with `requireAdmin`, not `requireTenant`.**
  STEP 0.5 confirmed the backend (`admin_pipeline.py::get_brand_identity`) hardcodes
  `tenant_id = "00000000-0000-0000-0000-000000000001"` (AA-internal sentinel) — despite the URL
  segment, this is not tenant-owned data. Documented inline so a future reader doesn't "fix" it back
  to `requireTenant` based on the path name alone.
- **Step 7 (middleware.ts matcher expansion) — skipped, not attempted.** See "Should know" below.
- **Content-role 401 through `tenant/[...path]` staff branch — kept as-is, not special-cased.**
  Grep confirmed content-role does reach this branch through 3 real pages (`/upload`, `/catalog`,
  `/admin/master-content` — see call sites below), so this is a real, not hypothetical, behavior
  change. Decision: keep the 401. Business reason — content-role is not in active operational use
  today; all upload/review/approve work is currently done by real admin accounts. The old
  cookie-only trust for `content` was the vulnerability this ticket exists to close, not a feature
  to preserve. If content-role is put into real operation later, it needs its own real
  authentication (a JWT-issuing login, not a bare `cis_role` cookie) — tracked as a follow-up, not
  solved here by carving out an exception that reintroduces the same unverified-cookie trust this
  diff removes.
  - Confirmed call sites affected: `frontend/app/(internal)/upload/page.tsx:185`
    (`/api/tenant/v1/pipeline/sources`), `frontend/app/(internal)/catalog/page.tsx:147,477`
    (`/api/tenant/v1/tours/{id}/approve`, `/api/tenant/v1/tours`),
    `frontend/app/admin/master-content/page.tsx:812` (`/api/tenant/admin/tenants/{id}/details`).
    Backend has no content-specific logic either (`verify_admin_secret` grants the same access
    regardless of admin vs. content) — the client-side role was never a real authorization
    boundary, only a cosmetic one.

## Changed

- `frontend/lib/auth-server.ts` (new) — `requireAdmin()` / `requireTenant()`.
- `frontend/app/api/admin/[...path]/route.ts` — added `requireAdmin()` gate; removed the
  decode-without-verify JWT helper and its stale comment (which incorrectly assumed middleware.ts
  had already verified the caller — it never runs for `/api/*`, see Should-know).
- `frontend/app/api/pipeline/[...path]/route.ts` — added `requireAdmin()` gate (route had zero auth
  check before this change).
- `frontend/app/api/tour-full/[id]/route.ts` — added `requireAdmin()` gate (same, zero check before).
- `frontend/app/api/tenant/pipeline/brand-identity/route.ts` — added `requireAdmin()` to both GET
  and POST (same, zero check before).
- `frontend/app/api/tenant/[...path]/route.ts` — staff branch (`isStaff`) now calls `requireAdmin()`
  instead of trusting the raw `cis_role` cookie; removed the now-unused `decodeJwtPayloadUnsafe`
  helper. Tenant branch untouched.

## Tradeoffs

- **Content-role sessions now get 401 through 2 routes they previously passed** (`tenant/[...path]`
  staff branch, and indirectly any page whose fetches go through it). Backend `/auth/verify-admin`
  only whitelists `role in ("admin","reviewer")`, and content-role logins never mint a JWT at all
  (separate `CONTENT_PASSWORD` path in `auth/login/route.ts`) — so there is no token `requireAdmin`
  could ever accept for that role. This is a real behavior change, not a bug in this diff: the *old*
  behavior was itself the vulnerability (any client could set `cis_role=content` by hand and get
  `X-Admin-Secret` forwarded with zero proof of identity). Flagged for product decision — see
  "Should know".

## Should know

- **Step 7 (middleware.ts matcher expansion) was intentionally skipped.** `PROTECTED_ROUTES` is a
  flat `{prefix, roles}` list keyed on *page* paths (`/admin/tenants`, `/admin/dashboard`, `/upload`,
  `/portal`, ...), each resolving to exactly one role set and one verify path
  (`verifyAdminToken` or `verifyTenantToken`, chosen via a single `cis_role` cookie read). None of
  the 4 target API prefixes (`/api/admin`, `/api/pipeline`, `/api/tour-full`, `/api/tenant`) have a
  matching entry, and forcing a fit surfaces three concrete mismatches:
  1. `/api/tenant/[...path]` proxies TWO different auth models behind one prefix — a tenant-JWT
     branch and a staff/admin-secret branch, selected by the route's own business logic
     (`cis_role`), not by a single role-to-token mapping the middleware model assumes.
  2. `/api/tenant/pipeline/brand-identity` is a sub-path of `/api/tenant/*` but requires
     `requireAdmin` only (see Decisions above), while sibling paths like
     `/api/tenant/v1/tours/pool` are legitimately tenant-accessible — one prefix rule can't express
     both.
  3. `/api/admin/[...path]` is a single catch-all proxy serving pages with different role
     requirements (`/admin/tenants` is admin-only; most others allow `content`) — the proxy itself
     doesn't distinguish which backend sub-resource is more sensitive, so a flat prefix rule would
     either over- or under-restrict relative to page-level policy.
  Route-handler-level gating (this diff) is authoritative and doesn't have this ambiguity — it reads
  the same cookie the *specific* route needs and calls the *specific* verify endpoint, per route.
  Middleware-level gating for `/api/*` would need per-endpoint entries mirroring the route-handler
  logic 1:1, which is real design work, not a "just add 4 lines to matcher" change — left to a
  follow-up if defense-in-depth at the middleware layer is wanted later.
- **Manual verify (task's step a/b/c/d):** step (a) — no-cookie → 401 — was run live against the
  local dev server for all 4 routes and passed (see PR description / chat transcript for curl
  output). Steps (b)/(c)/(d) — valid admin JWT / browser UI regression / tenant JWT — were **not**
  completed in-session. No admin/tenant test credentials were available (correctly not stored in
  the repo or Secrets Manager — checked `aws secretsmanager list-secrets`, only found the retired
  `ADMIN_SECRET`, no admin username/password). Per plan: pushed to a PR against `develop` for
  Nghiep to UAT live on the Vercel Preview build (real browser login, real admin account) before
  merging — this is the actual completion of steps (b)/(c)/(d), not skipped, just moved to a
  environment where real credentials are available.
- `npm run lint` shows 172 pre-existing problems (107 errors, 65 warnings) — all in files untouched
  by this diff (react-hooks rules on `admin/tenants/page.tsx`, `admin/upload/page.tsx`, etc.).
  `npm run build` compiles clean; all 6 changed/added files show zero lint findings.
