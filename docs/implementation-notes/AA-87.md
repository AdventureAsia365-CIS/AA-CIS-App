## AA-87: S1 Brand Injection Fix

### Decisions
- Decision: brand_system_prompt already flows into LLMRequest.system_prompt via generate_node (graph.py:62-68). The fix adds the missing observability and unbranded flagging, not a new wiring path.
- Decision: store unbranded flag in `og_tags JSONB` (`{"unbranded": true}`) rather than a new column — avoids migration cost, og_tags is unused and defaults to `{}`.
- Decision: log prompt_len in generate_node (not in _call_bedrock) so it's visible per LangGraph invocation, not per Bedrock retry.
- Decision: `is_branded` added to ContentState TypedDict and returned through `_rewrite_tour` into `run_tour` INSERT.

### Changed
- graph.py: Added `is_branded: bool` to ContentState, prompt_len structlog entry in generate_node.
- v1_pipeline.py: `_rewrite_tour` now returns `is_branded`; `run_tour` INSERT includes `og_tags` param set to `{}` (branded) or `{"unbranded": true}`.
- test_content_graph.py: Added `test_generate_node_injects_brand_system_prompt` — mocks LLMClient.generate, asserts LLMRequest.system_prompt contains brand text.

### Tradeoffs
- og_tags reuse: semantically impure but avoids a migration. If og_tags ever gets used for real OG data, this flag survives (it's a JSONB merge, not a replace). Acceptable for now.
- prompt_len logging adds ~20 bytes per CloudWatch log event. Negligible at current volume.

### Should know
- brand_system_prompt can be empty string (tenant has no brand config) — this is valid and should NOT raise. Only log warning + set unbranded flag.
- The brand content is appended to SYSTEM_PROMPT, not replacing it, so the AA core rules always apply.
- prompt_len reflects the FULL system including AA base + language + brand append.
