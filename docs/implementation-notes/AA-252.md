# AA-252 — Admin auth bypass (middleware trusts unsigned `cis_role` cookie) — Implementation Notes

## Context
Discovered while manually testing AA-250 B1 in the browser: had to bypass the login page by
hand-setting `document.cookie = "cis_role=admin"` in Playwright to reach `/admin/review`, and it
worked with zero credentials. Investigating why turned into this fix.

## Decisions
- **Rewrote `middleware.ts` from an exclude-list to an allow-list.** The pre-existing shape
  (`ADMIN_PATHS`/`INTERNAL_PATHS`/`TENANT_PATHS` arrays, each with bespoke pass conditions) only
  blocks what it explicitly checks for — anything a check forgets to cover is open by default.
  Replaced with a single `PROTECTED_ROUTES: { prefix, roles }[]` table; a path with no entry, or a
  role not in that path's list, is denied by default (`redirect("/login")`).
- **`content` role left as-is (explicit scope decision).** It has no JWT/token and nothing
  re-verifies it after the initial `CONTENT_PASSWORD` check at login — the `cis_role` cookie is
  trusted at face value, unchanged from before this fix. Redesigning it was ruled out of scope;
  tracked under AA-253 so it isn't mistaken for solved.
- **BFF proxy routes (`/api/admin/*`, `/api/pipeline/*`, `/api/tour-full/*`,
  `/api/tenant/pipeline/*`) are NOT covered by `middleware.ts`'s `config.matcher`** — confirmed by
  direct `curl` with zero cookies returning `200` with real data. This is a distinct, deeper gap
  (middleware can only gate pages, never the API routes browsers call) — filed as AA-253, not
  fixed here.

## Changed
- `frontend/middleware.ts` — full rewrite (allow-list model), closes 5 bypasses found during
  investigation, all rooted in trusting `cis_role` (a plain, non-httpOnly, client-writable cookie)
  without a corresponding server-side verification:
  1. `verifyAdminToken()` treated a **missing** `cis_admin_token` as "pass" (an interim carve-out
     for the legacy ADMIN_SECRET fallback session shape). Now fails closed.
  2. `INTERNAL_PATHS` only ran `verifyAdminToken()` when `role === "admin" || "reviewer"` — any
     other string (e.g. `cis_role=banana`) skipped verification and passed straight through.
  3. `TENANT_PATHS` special-cased `role === "admin"` to `return NextResponse.next()` immediately,
     bypassing `verifyAdminToken()` entirely — independent of bug #1's fix.
  4. `/admin/run-health`, `/admin/s1-rewrite`, `/admin/settings` are real pages never listed in
     `INTERNAL_PATHS`/`ADMIN_PATHS` — fell through to the unconditional `NextResponse.next()` at
     the end of the function, zero auth, even with no cookies at all.
  5. `/brand` (`app/(internal)/brand`, a route group — not `/admin/brand`) was missing from
     `config.matcher` entirely, so middleware never executed for it.
- `frontend/app/api/auth/login/route.ts` — retired the legacy `ADMIN_SECRET` shared-secret
  fallback (the session shape bug #1 above existed to accommodate). Verified via direct DB query
  (read-only, via S3-mediated ECS exec) that both real admins (`nghiep`, `admin`) already have
  JWT accounts in `shared.admin_users` — the fallback served no one. On JWT infra failure
  (network/timeout/5xx) the route now returns a real error instead of silently granting an
  unauthenticated session.
- `frontend/app/login/page.tsx` — **NOT changed.** Original plan (from the interrupted task
  script) was to delete the `document.cookie = cis_role=...` line here too, on the assumption the
  JWT flow doesn't depend on it. That assumption was checked and is **false**: `cis_role` is what
  `middleware.ts` reads to decide which `PROTECTED_ROUTES` entry applies and whether to call
  `verifyAdminToken()`/`verifyTenantToken()` at all — removing it would lock out real JWT logins.
  Left in place; `cis_api_token`/`cis_user` (also set there) are UI-display-only, not read by any
  middleware check, so leaving them is inert either way.

## Tradeoffs
- **`content` role stays unhardened** (see Decisions). Anyone who forges `cis_role=content` still
  reaches the same internal pages a real content-login session would — no new exposure (identical
  to pre-fix behavior), but also not fixed. AA-253.
- **BFF proxy routes stay unauthenticated at the route level** (rely entirely on `ADMIN_SECRET`
  being attached server-side, with zero check of the calling browser session). Confirmed via
  direct `curl` (no cookies) against `/api/admin/review-queue` on the dev server → `200` with real
  data, unaffected by this PR since these routes were never in `middleware.ts`'s matcher to begin
  with. AA-253 — a materially larger fix (route-level JWT checks instead of a single page-gate),
  deliberately not bundled into this PR to keep the security fix reviewable and shippable today.

## Should know (before reading the diff)
- No FE unit/integration test suite exists for `middleware.ts` (no jest/vitest config, no test
  script in `frontend/package.json`, `find frontend -iname "*.test.*"` empty). Verified entirely
  by hand: `curl` against every closed bypass (with the exact cookie values that exploited them
  pre-fix) confirms `307 → /login` (or `/tenant-login` for `/portal`); `npm run build` (TS)
  green; and Nghiep manually logged in as both real admin accounts (`nghiep`, `admin`) in a
  browser against the local dev server and confirmed `/admin/dashboard` loads normally — the real
  JWT path is unaffected.
- `tests/unit/` (backend, 495 tests) run for completeness even though this PR touches
  frontend-only files — 0 regression, as expected.
- Backend endpoints touched by verification only (not modified): `/auth/admin-login`,
  `/auth/verify-admin` — both confirmed live and returning `401` for bad credentials/garbage
  tokens (not `404`/`500`), via direct `curl` against the real Dev API.
- Real `shared.admin_users` row count at time of this fix: 2 (`nghiep`, `admin`), both
  `role='admin'`, both `is_active=true` — checked directly via read-only DB query (S3-mediated ECS
  exec), not assumed from prior session notes.
