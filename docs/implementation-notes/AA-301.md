# AA-301 — N4 Runway Map / N5 Quarter Plan / N6 Slot Allocator

## Decisions

- **Repo/module location corrected**: the original issue text said "Repo: AA-ACP-App", but
  AA-ACP-App is a pure Next.js/TypeScript frontend (no Python at all). N0-N2 (AA-299/AA-302,
  atom decompose) actually live in **AA-CIS-App** (`api/routers/v1_atoms.py` +
  `services/acp_shared/`). N4/N5/N6 built here as `services/acp_planning/`, following the
  one-package-per-stage convention already established by `services/acp_s3/`,
  `services/acp_s4_social/` (small files split by concern + local docstrings, not one big
  file like the aamc research prototype).
- **Data source: real DB, not a fixture.** `acp_contract.v_trip_registry` (migrations
  078/080/083) + `acp_contract.tour_atoms` (migration 079). Neither exposed `tenant_id` before
  this issue — migration 087 adds `rt.tenant_id` + `rt.lifecycle_stage` to the view (purely
  additive; sole consumer `v1_atoms.py` uses named-column SELECTs, verified live it still
  works post-migration).
- **Multi-tenant is mandatory, `tenant_id: UUID` is a required first parameter** on every
  public N4/N5/N6 function — no implicit session/context read. Verified live:
  `raw_tours.tenant_id` is a real UUID FK'd to `shared.tenants` (11 tenants registered, 7
  active) but 793/793 current rows belong to a single tenant (`aa_internal`,
  `00000000-0000-0000-0000-000000000001`) — no other tenant has real tour data yet, so
  multi-tenant isolation is tested with an in-memory fixture second tenant
  (`tests/unit/test_aa301_multitenant.py`), never inserted into the dev DB.
- **`tour_atoms` gets no `tenant_id` column** — confirmed live it has none today (16→18
  columns, `owner_scope='platform'` for all 235 live atoms), matching the D3 principle
  (atoms curated once, shared across tenants). Tenancy is inherited via
  `tour_atoms.tour_id -> raw_tours.tenant_id` (JOIN), not a direct column.
- **`services/acp_s3/planner.py`'s `funnel_mix` is explicitly OUT of scope** for this issue —
  investigated in a prior turn: `acp_s3` (ACP v1, AA-45) operates on `tenant_id + run_id +
  country` with an LLM-driven (Bedrock Sonnet) calendar with no `trip_id`/`tour_id` concept
  anywhere in its models; N4-N6 operate on CIS's own `raw_tours`/`tour_atoms`. There is no
  live mapping between an ACP-B2B tenant's "country" and specific CIS `tour_id`s (`acp_runs`/
  `acp_run_context` are both empty, 0 rows, in dev). Reconciling the two is a real
  architecture decision for a separate ticket, not silently bridged here.
- **Migration 086 gap fixed**: `raw_tours.lifecycle_stage` (`active|phasing_out|retired`)
  was applied live to the dev DB in a prior session via a one-off script, but the `.sql` file
  was never committed. Written retroactively (idempotent guards, not re-executed since
  already live) so migration history matches DB reality.

## Tradeoffs / bugs ported and fixed (not carried over)

All 6 bugs (B4/B5/B6/B7/B9/B11) were confirmed as **real, pre-existing code** in
`aamc/planning.py` (the research prototype), not hypothetical — verified against the exact
line numbers before porting. Fixed during the port, not copied:

- **B9** (family detection): scanned `trip.summary`, which doesn't exist pre-publish in this
  schema (`v_trip_registry` only exposes `pt.aa_summary`, NULL until publish). Fixed to scan
  `itinerary_source` (`rt.src_itineraries`, always populated from S0). Also fixed: destination
  offset used `trips[0]` as a "representative" trip — now aggregated (min lo, max hi) across
  every trip at that destination.
- **B11** (MOFU band): root cause was structural, not a constant tuning issue — a
  destination's achievable "distance to nearest season" is capped by season-spacing geometry
  (~5-6 months for typical 2-season destinations) and the old MOFU band sat entirely above
  the market's own `hi` (6 for long_haul), so for common geometries MOFU was mathematically
  unreachable. Redesigned: BOFU keeps the inner half of `[lo, hi]`, MOFU takes the outer half
  plus one more band-width beyond `hi`. Verified against the issue's own Sapa numbers
  (`starts=[3,9]`, `lo=3,hi=6`) — MOFU now appears (2/12 months), matching the required
  verify criterion.
  - **Known follow-up not fixed here**: the same structural issue recurs for
    `family_extended` (lo=6, hi=12) — its `lo=6` already exceeds the typical max achievable
    distance (~5-6), so a family-labeled trip at a common 2-season destination gets 100%
    TOFU, 0% BOFU/MOFU, under the *new* formula too (caught by a self-inflicted test bug
    during implementation — my own fixture text accidentally contained the substring
    "family", which correctly triggered the detector and exposed this). Fixing it would mean
    changing `RUNWAY_OFFSETS_MONTHS["family_extended"]` itself (a market-tuning constant, not
    a band-formula bug) — flagged for a follow-up ticket, not changed unilaterally.
- **B4** (special matching): substring containment (`"sapa trekking" in "sapa valley trek"`)
  both false-negatived the issue's own example and could false-positive on partial words.
  Replaced with token overlap + 4-char prefix fuzzy match, threshold 50% of the special's own
  tokens matched.
- **B5** (thin-trip cap): `THIN_TRIP_ATOM_MIN` existed in config but nothing capped share by
  atom count anywhere in the research build (grep-confirmed). Added `_cap_thin_trip_shares()` —
  caps at `THIN_TRIP_MAX_SHARE` (0.15, **self-chosen**, not specified by the issue) and
  redistributes freed share proportionally to non-thin destinations (redistribution behavior
  also self-chosen).
- **B6** (per-trip keyword): the real bug lived one layer downstream of N6 in the research
  build (`research.py::compile_brief()`/`keyword_research()`, module C, not yet built here).
  Scope for AA-301: `Slot.keyword_seed` is populated per-slot from that slot's own top chosen
  atom — since B7 guarantees no atom repeats within a month for the same trip+channel,
  keyword_seed is naturally distinct slot-to-slot without needing the C module.
- **B7** (in-month cooldown): cooldown was only enforced post-publish
  (`atom.cooldown_until`), so repeated slot calls within one month picked the same top-N
  atoms. Added a `used_this_month: dict[(trip_id, channel), set[atom_id]]` tracked across the
  whole `compute_slot_grid()` call.
- **Atom floor (AA-300, not a bug fix, a new requirement)**: falls out of the same
  used-this-month tracking — an exhausted pool drops the slot (never repeats) and logs a
  note. Found and fixed a real bug in my own first implementation here: the log list wasn't
  deduped, so one exhausted destination could log the identical line 60+ times in one call
  (the round-robin retries it many times, matching the original code's intentional "grid
  stays short" behavior) — added `_add_note()` dedup helper.

## Self-chosen decisions requiring review (not specified in the issue text)

1. **MOFU band formula** (B11): inner/outer half-split of `[lo, hi]` + one extra band-width
   for MOFU, two more for TOFU. See `runway.py::_stage_for_dist()` docstring for the full
   derivation against the issue's own Sapa numbers.
2. **Duration parser** (`runway.py::parse_duration_days()`): built from a live survey of 749
   non-null `duration_raw` values (not the 2 examples in CLAUDE.md) — `# days`/`# DAYS`/
   `# day(s) # night(s)` extract the day count; `HALF DAY`/`DAY TRIP`/`same day`/hour-only/
   `# MIN.` all map to 0 (same-day, no overnight); `OVERNIGHT TRIP` maps to 1; unparseable
   returns `None` (never fabricated).
3. **THIN_TRIP_MAX_SHARE = 0.15** and proportional redistribution of freed share to non-thin
   destinations (quarter.py).
4. **Gate B implementation**: `QuarterPlan.approved`/`approved_by` fields +
   `QuarterPlanNotApprovedError` raised by `allocate_month()`/`compute_slot_grid()` — does
   NOT reuse `acp_shared.acp_hitl_requests` (that table is FK'd to `acp_shared.acp_runs.run_id`,
   an ACP-B2B "pipeline run" concept N4-N6 don't have).
5. **`markets` / `capacity_posts_per_week` / `channels` / `specials` / `primary_market` are
   all required caller-supplied parameters**, not read from any tenant-config table — no such
   table exists yet in this schema (the aamc research build read these from a `decisions`
   store that has no equivalent here). Flagged as a real gap for a follow-up ticket
   (tenant campaign/brand config), not silently invented.
6. **phasing_out interpretation**: the issue's recompute-trigger text ("Tour phasing_out ->
   N6 tháng hiện tại") is ambiguous between "recompute now" and "only allocate the current
   month". Implemented as: a `phasing_out` trip is excluded from any month's `SlotGrid` other
   than the real current calendar month (`date.today()`), logged when excluded.
7. **Test location**: `tests/unit/test_aa301_*.py`, not `services/acp_planning/tests/` —
   `services/acp_planning` is imported directly into the main app (relative imports, like
   `services/content_generation`), which is tested from top-level `tests/unit/`; the
   package-local `tests/` convention (`acp_s3`, `acp_s4_social`) is only used by services
   that are separately Lambda-deployed with flat imports.
