# AA-186 — Gate-as-event-boundary wiring + acp.hitl.approved contract

## Decisions

- **Payload builder lives in a new module `services/acp_shared/hitl_events.py`**
  (not inlined in `event_constants.py`, which is constants-only per its own
  docstring). Exposes `build_hitl_event_payload()` and `publish_hitl_event()`,
  used identically from all 3 publish sites.
- `gate = stage - 1` (generic formula: stage 2→gate 1, stage 3→gate 2,
  stage 4→gate 3). `next_stage` comes from `NEXT_STAGE_BY_GATE = {1: 3, 2: 4, 3: None}`,
  applied only when `decision == "approved"` (always `None` on reject).
- `reviewer_type` in the event payload = `tenant.get("reviewer_type")` from
  `verify_tenant_api_key()` (`"aa_internal" | "tenant_self"`). This is a
  **different namespace** from `acp_hitl_requests.reviewer_type`
  (`"aa_internal"|"tenant_admin"`, CHECK-constrained, migration 036) and from
  `_audit_actor_type()`'s `audit_log.actor_type` enum
  (`"hitl_reviewer"|"tenant_admin"`). Neither of those is read for the event
  payload — documented inline at each publish call site.
- `publish_hitl_event()` never raises — logs and returns `False` on any
  boto3/EventBridge error, matching the existing `v1_s3.py` gate2 pattern
  (approve/reject HTTP response must not fail just because the event bus is
  unreachable).

## Changed vs. original ticket text

- Scope locked to **gate=1 (stage=2) and gate=2 (stage=3) only** (per STEP 0
  resolution). `/v1/acp/gate/s4/approve` (stage=4/gate=3) remains 404 — no
  `acp_hitl_requests` row with `stage=4` is ever created (S4's HITL review
  uses a separate per-draft mechanism: `acp_silver_s4.blog_drafts.hitl_gate3_status`,
  unrelated table/flow). The `gate = stage - 1` formula is generic and would
  produce `gate=3 → next_stage=None` correctly if ever reached, but no
  producer for that path was added.
- `v1_s3.py`'s legacy `/v1/hitl/gate2/{run_id}/approve|reject` (queries
  `stage=2`, old JWT/`X-Admin-Secret` auth, already publishes a non-conformant
  `acp.hitl.approved` payload) was **left untouched** per scope lock — it's
  effectively dead code (its `stage=2` filter never matches `_create_hitl_gate2`'s
  `stage=3` inserts). Retirement tracked separately.
- 3 publish points implemented exactly as scoped:
  1. `v1_acp_gate.py::gate_approve` → `acp.hitl.approved`
  2. `v1_acp_gate.py::gate_reject` → `acp.hitl.rejected`
  3. `services/acp/s2/router.py::_handle_gate1` (auto-approve path) → `acp.hitl.approved`

## Tradeoffs

- `_handle_gate1`'s existing `INSERT ... ON CONFLICT DO NOTHING` was extended
  with `RETURNING hitl_id` so the publish/awaiting_gate branch only fires when
  a row was actually inserted (not on a no-op due to a prior conflict). The
  underlying conflict-target correctness of that `ON CONFLICT DO NOTHING` is
  pre-existing and out of scope.
- `acp_stage_runs.status = 'awaiting_gate'` is set at HITL-row-creation time
  in both `_handle_gate1` (pending/non-auto-approve path, stage='s2') and
  `_create_hitl_gate2` (always, stage='s3'). Clearing/transitioning this
  marker on approve is **not** done here — that belongs to the next-stage
  trigger consumer (AA-187), which is the natural place to flip
  `acp_stage_runs.status` when it picks up `acp.hitl.approved` and starts the
  next stage.
- Idempotency: producer-side double-publish is prevented by the existing
  `status != 'pending' → 409` guard (publish only happens after a successful
  pending→approved/rejected transition, so a second call 409s before
  reaching the publish call). Consumer-side dedup (`acp_stage_runs.event_id`
  partial-unique index, AA-106) is reused as-is — no changes needed, AA-187
  territory.

## Should know

- No migration — all columns used (`acp_hitl_requests.*`, `acp_stage_runs.status`,
  `acp_stage_runs.event_id`) already exist (migrations 036, 056).
- `acp_hitl_requests.expires_at`, queried by `get_gate_run`/`get_s3_run`, isn't
  defined in any migration file (likely applied via ECS exec pre-migrations,
  same as `idempotency_keys` per migration 028's note) — untouched, not part
  of this change.
- New tests: `tests/acp/test_aa186_hitl_events.py` (5 tests) — approve/reject
  payload shape, 409-no-publish, and both Gate 1 auto-approve / pending
  branches of `_handle_gate1`.
- 8 pre-existing failures in `tests/acp/test_h3_rule_extractor.py`,
  `test_s2_idempotency.py`, `test_s2_semaphore.py` are unrelated to this
  change (verified via `git stash` — fail identically on `develop`).
