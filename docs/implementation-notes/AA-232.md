# AA-232 — Per-user admin JWT auth (`shared.admin_users`) — Implementation Notes

## Decisions
- **App-code-first sequencing.** Discovery (STEP 0/0b/0c) surfaced that the live AA-241
  review-edit endpoint writes a free-text handle into `generated_content.reviewed_by`
  (`admin_pipeline.py:2981`). Migration 074 retypes that column to `UUID`, so applying the
  migration against the running app would 500 every review edit
  (`invalid input syntax for type uuid: "admin"`). Chose to patch the app **before** applying DDL.
- **Interim reviewer identity = NULL, handle preserved in `edit_diff`.** `admin_users` is created
  empty (no seed) and per-user admin JWT login is not built yet, so there is no verified
  `admin_users.id` to write. The patch writes `reviewed_by = NULL` and stores the unverified
  handle under `edit_diff.reviewer_handle` (jsonb — type-safe on both sides of the migration).

## Changed
- `api/routers/admin_pipeline.py` (~L2979–2996, `PATCH .../generated/{content_id}`):
  - `reviewed_by = $N` (bound free-text handle) → `reviewed_by = NULL` (SQL literal).
  - `edit_diff` payload `{"fields": [...]}` → `{"fields": [...], "reviewer_handle": "<x-reviewer-id or 'admin'>"}`.
  - Dropped the `reviewer` bind param; renumbered placeholders (edit_diff/id/tour_id shift down by one).
- Migration file `api/migrations/074_admin_users_jwt_auth.sql` authored by Nghiep (not applied yet).

## Tradeoffs
- **Audit regression (accepted, interim):** the review-queue GET (`admin_pipeline.py:1911`) now
  returns `reviewed_by = NULL` for new edits instead of the handle. The "who" is retained in
  `edit_diff.reviewer_handle` until real per-user JWT identity lands and writes a verified UUID.
- **Why NULL over a validated UUID:** post-migration `reviewed_by` FKs to `admin_users`; any
  non-null UUID not present in the (empty) table would be a FK violation. NULL is the only value
  safe across both the current `varchar` and post-migration `uuid` column type.

## Part 2 — admin JWT auth endpoints (S86)
### Decisions
- **bcrypt password login, not sha256 api_key** (ADR-2026-017): admins get real username+password
  login UX, deviating from the tenant `api_key_hash` (sha256) pattern. `+bcrypt>=4.1.0` in
  requirements (image resolves 5.0.0). Helpers/models (`_verify_password`, `_create_admin_jwt`,
  `AdminLoginRequest/Response`, `VerifyAdminResponse`) live in `api/routers/auth.py`.
- **Endpoints declared in `main.py`, not via router mount.** Discovery: the `/auth` `router` object
  in `auth.py` is **never `include_router`'d** — main.py imports helpers and re-declares each auth
  route inline with `@app.post(...)` + `Depends(get_pool)` (that's how `tenant-login`/`verify-tenant`
  work). My first pass put the endpoints on the unmounted router → live **404** (caught by smoke test).
  Fixed by declaring `/auth/admin-login` + `/auth/verify-admin` in `main.py`; **stripped the dead
  `@router.post` admin functions** from `auth.py` (kept only helpers+models — one implementation).
- **Admin vs tenant JWT share `JWT_SECRET`/HS256/`verify_jwt()`**; distinguished by claims:
  admin `sub = admin_users.id (UUID)`, `role in ('admin','reviewer')`; tenant `role = 'tenant'`.
  `verify-admin` uses an **explicit role whitelist** (`not in ('admin','reviewer')` → 401) so a
  tenant token (which decodes fine under the shared secret) can't pass the admin verify endpoint.

### Tradeoffs
- **Constant-timing login:** `admin-login` always runs a bcrypt compare (dummy hash on no-row) and
  returns an identical 401 for unknown-user / wrong-password / inactive — no enumeration signal.
- **Stateless verify:** `verify-admin` does not re-check `is_active` per call (same trade-off as
  `verify-tenant`) — a deactivated admin stays valid until token expiry (`JWT_EXPIRY_H = 24h`).

### Verified (S86, live on `api:345`)
- `/auth/admin-login` bogus user → **401** (not 500 → bcrypt path OK; not 404 → mounted);
  empty fields → **400**; `/auth/verify-admin` bad token → **401**. `shared.admin_users` still empty
  (real rows inserted by hand later), so no success-path row yet.
