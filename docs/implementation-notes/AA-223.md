# AA-223 тАФ Async run-tour (202 + job poll)

ADR-2026-016. Decouple `POST /admin/run-tour` from the full synchronous pipeline:
a new async endpoint inserts a `queued` job row, dispatches the existing
`_run_tour_safe` executor in the background, and returns `202` + a poll URL. A
new `shared.pipeline_jobs` table tracks each job through
`queued тЖТ running тЖТ succeeded | failed | interrupted`.

Files: `api/migrations/071_pipeline_jobs.sql`, `api/routers/jobs_repo.py`,
`api/routers/admin_pipeline.py`, `api/main.py`,
`tests/unit/test_aa223_async_run_tour.py`.

## Decisions

1. **Migration 071 PK + FK columns are UUID, not BIGINT.** The Step-1 template
   proposed `result_version_id BIGINT` / `pipeline_run_id BIGINT`. Verified against
   the real schema: `silver_aa_internal.generated_content.id` (003:385) and
   `shared.pipeline_runs.id` / `batch_id` (003:266-268) are all `UUID DEFAULT
   gen_random_uuid()`. So `result_version_id` and `pipeline_run_id` are `UUID`.
   `_execute_run_tour` returns `"version_id"` already `str()`-cast тЖТ fits UUID cast.

2. **`result is None тЖТ mark_failed` guard.** `_run_tour_safe` (semaphore + 3├Ч retry)
   SWALLOWS the final failure: when all attempts raise it logs, marks the
   `pipeline_runs` row failed, and `return`s **None** тАФ it never re-raises. So the
   `except` in `_run_tour_job` would never fire on a normal retries-exhausted
   failure. Without a guard a hard-failed tour would be marked `succeeded`
   (version_id NULL). `_run_tour_job` keys on `result is None тЖТ mark_failed`, which
   is the exact, unambiguous signal of the swallowed hard-fail.

3. **`pipeline_run_id` is always NULL in this phase.** `_execute_run_tour`'s return
   dict has `"version_id"` and `"batch_id"` but **no** `pipeline_run_id` key. Rather
   than invent a value, `mark_succeeded(job_id, version_id, None)` writes NULL. The
   column exists for a later wiring; not fabricated now.

4. **GC-safe background-task set.** `asyncio` keeps only a weak ref to scheduled
   tasks тАФ a fire-and-forget `create_task` can be GC'd mid-flight, leaving a job
   stuck `running` forever (looks like a Fargate restart but isn't). Fix: module-level
   `_background_tasks: set`, `add(task)` + `task.add_done_callback(_background_tasks.discard)`.

5. **Auth match 1:1.** Both new endpoints (`POST /run-tour-async`, `GET /jobs/{job_id}`)
   copy `/run-tour`'s guard exactly: `x_admin_secret: str = Header(None)` +
   `verify_admin_secret(x_admin_secret)`. No new auth surface left open.

6. **New endpoint, sync `/run-tour` kept.** `/run-tour-async` is additive;
   `/run-tour` stays synchronous for backward-compat. FE / verify-scripts migrate to
   the poll model incrementally (no big-bang). The `202` + `poll_url` contract is
   stable across a future SQS upgrade (ADR "Future" = H╞░с╗Ыng Y), so the swap stays
   transparent to clients.

## Changed (vs. the Step plan)

- Template column types `BIGINT` тЖТ `UUID` (Decision 1).
- Wrapper guard added beyond the bare `try/except`: `result is None тЖТ mark_failed`
  (Decision 2) тАФ the plan flagged this as conditional on the swallow finding; it is
  required.
- `_run_tour_safe` signature `-> None` тЖТ `-> dict | None`; success path
  `await _execute_run_tour(...); return` тЖТ `return await _execute_run_tour(...)`
  (1-line behavioural change; all prior callers discarded the return value).
- Dedup response also includes `poll_url` (plan template omitted it on the dup
  branch) тАФ both 202 branches now carry it for client uniformity.

## Tradeoffs

- **Soft-fail = succeeded.** A returned dict with `version_id=None` (version insert
  returned null) is marked `succeeded` + `result_version_id` NULL, not `failed`. The
  pipeline ran to completion; client infers soft-fail from the null version. Marking
  it `failed` would conflate "ran but produced nothing" with "crashed after retries".
- **Startup sweep is best-effort.** `sweep_interrupted()` is wrapped in try/except in
  the lifespan; a transient DB error at ECS cold-start must NOT crash boot
  (crash-loop risk). Worst case an orphaned job stays `running` one extra cycle.
- **Own asyncpg.connect per call (no app pool)** in `jobs_repo` тАФ matches
  `_execute_run_tour`'s existing pattern for uniformity over pool efficiency.
