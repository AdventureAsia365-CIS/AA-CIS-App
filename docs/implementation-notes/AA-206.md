# AA-206 — S1 Two-Model Generate–Judge (Bedrock writes, GPT-4.1 judges)

ADR: ADR-2026-014. Linear: AA-206. Branch: `pqnghiep1354/aa-206-...generatejudge-bedrock-viet-gpt-41-cham`.

## Goal
Add GPT-4.1 as a brand-fit **judge** inside the S1 content-generation graph. Bedrock writes,
GPT scores + gives feedback (does NOT edit content). Bedrock fixes via the existing retry loop
(`quality_score` vs `MIN_QUALITY` + `feedback`). Also strengthen the brand differentiation block.

## Decisions
- **Step 1 (diff_block):** CONTRAST REQUIREMENT widened from "summary and highlights" →
  "summary, highlights, itineraries (incl. each day-title), and the overall framing". Added a
  negative-contrast paragraph showing generic phrasing to avoid + an instruction to lead with the
  brand's specific mission angle (no synonym-swap). Kept the early `return ""` guard so legacy/default
  brands (no differentiation signals) still get an empty block — backward compatible.

- **Step 2 (judge_node):** New `services/content_generation/judge_node.py::judge_node`.
  - Scores via `LLMClient(model_tier="gpt-4.1")` (no new client; no Bedrock fallback path for gpt-4.1).
  - `quality_score = min(validate_score, judge_score)` — judge gate **stacks on** validate's structural
    gate; high brand-fit can't mask a structurally broken output, and vice-versa.
  - `judge_score = min(brand_fit, cross_brand_distinct)`; `mission_present=False` caps at 6.0 (below
    MIN_QUALITY=7.0) to force ≥1 retry.
  - Feedback merged into `feedback` only when below threshold (a retry will actually fire); validate
    feedback preserved so Bedrock sees both signals.
  - **Skips** (returns state unchanged) when no generated content OR no brand differentiation profile
    (same signal guard as `_build_brand_diff_block`) — backward-compatible + no GPT cost for legacy brands.
  - All failures non-blocking: any GPT/parse error keeps validate's score, never raises.

- **Step 3 (graph wiring + brand_audit):**
  - Graph: `validate → llm_judge → should_retry{done→brand_audit, retry→increment_retry, hitl→END}`.
  - `brand_audit_node`: when `judge_brand_fit` present in state → `_audit_from_judge` (reuse judge
    result, keep deterministic `pre_audit_checks` for flag_fix targeting, surface judge feedback as an
    issue, **no second GPT call**). Falls back to the original standalone OpenAI audit only when the
    judge did not run (legacy/no-profile brands).

## Changed
- (Step 1) `services/content_generation/graph.py::_build_brand_diff_block` — widened CONTRAST + negative contrast.
- (Step 2) NEW `services/content_generation/judge_node.py`.
- (Step 3) `graph.py`: `ContentState` += judge fields; import + node `llm_judge`; edges rewired.
- (Step 3) `brand_audit_node.py`: `_audit_from_judge` + judge-present branch.

## Tradeoffs
- (Step 1) Added static negative-example text to the system prompt → +~300 chars per generation.
  Acceptable: prompt-cached on Bedrock and the contrast gain is the whole point of AA-202/AA-206.
- (Step 3) brand_audit reusing judge results means flag_fix loses the *LLM-derived* `fields_to_fix`
  for branded tours; it still gets deterministic pre-audit codes + validate SEO codes. Net: one fewer
  GPT call per branded tour, retry loop already enforced brand-fit before flag_fix runs. Legacy brands
  unchanged (still full OpenAI audit).

## Should know
- `_build_brand_diff_block` and `_build_judge_prompt` are pure string assembly → unit-testable w/o LLM/DB.
- Existing AA-202 tests assert exact substrings ("CONTRAST REQUIREMENT", field labels) — kept intact.
- judge_node imports `LLMClient`/`LLMRequest`; tests patch `services.content_generation.judge_node.LLMClient`.
- MIN_QUALITY (7.0) duplicated as `_MIN_QUALITY` in judge_node to avoid circular import (graph imports judge).
