# AA-215 — Revalidate node after flag_fix (POST-fix verification)

## Decisions
- `revalidate_node` re-runs `validate_node` + `judge_node` in the same order as the main graph
  (validate sets quality_score that judge then `min()`s against). Calling judge alone would min
  against the stale pre-fix validate score.
- Does NOT re-run `brand_audit_node` — judge is treated as the post-fix brand gate. Pre-fix
  deterministic audit codes already drove flag_fix's targeting.
- POST-fix outcome rewrites `brand_audit_status`: pass → `"fixed"` (publishable via AA-211 gate,
  which treats `flagged`+`fix_pass_applied` as publishable and only blocks `manual_check` /
  `flagged`-unfixed). fail → `"manual_check"` (blocked + routed to HITL by existing gate).
- Passthrough when `fix_pass_applied=False` (sets `revalidate_ran=False`, leaves status untouched).

## Changed
- graph.py: ContentState +2 keys (revalidate_ran, revalidate_passed); new revalidate_node;
  edge flag_fix→END replaced with flag_fix→revalidate→END.

## Tradeoffs
- "fixed" is a NEW brand_audit_status value (was "pass"/"flagged"/"manual_check"/""). AA-211
  `_is_publishable` already publishes anything that isn't manual_check / flagged-unfixed, so "fixed"
  is publishable without changing the gate. Downstream readers that switch on exact status strings
  may need to learn "fixed" (catalog readiness CASE in admin_pipeline already treats
  flagged+fix_pass_applied as publishable separately).

## Should know
- revalidate re-runs validate (free, rule-based) + judge (GPT-4.1, only fires for branded tours
  with a differentiation profile — judge_node self-skips otherwise, so no extra GPT cost on
  legacy/no-profile tours).
