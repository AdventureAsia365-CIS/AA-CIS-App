# AA-125 — pgvector Dedup Threshold Calibration

## Decisions

**Method used: SYNTHETIC TF-IDF (not real DB data)**

DB is stopped between sessions; `acp_silver_s4.blog_drafts` has no rows with
`content_embedding` populated at time of calibration (engine is new, no production
blog drafts published yet).

TF-IDF was chosen over sentence-transformers because sklearn is not installed in
the project environment (not in requirements.txt). Pure-numpy TF-IDF implemented
inline in the script — no additional dependencies.

**Decision: KEEP threshold at 0.92**

Despite TF-IDF benchmark recommending 0.80, the production system uses Bedrock
Titan Embed Text v2 (1536-dim dense semantic vectors). These are fundamentally
different from TF-IDF (sparse lexical). The 0.92 value comes from PRD v1.0 Q13
and was calibrated for Titan's embedding space. Changing it based on TF-IDF data
would be a category error.

## Why TF-IDF ≠ Titan Embed v2

TF-IDF cosine similarity measures lexical overlap (shared words, weighted by rarity).
Two blog posts can be semantically identical but use different vocabulary — TF-IDF
gives them low similarity while Titan collapses them to cosine > 0.95.

Conversely, TF-IDF gives high similarity when the same words appear, regardless of
meaning. Titan's 1536-dim embedding space compresses entire passages into dense
vectors where semantic distance is measured, not term overlap.

**At 0.92 in Titan's space:** two embeddings that close share essentially the same
semantic content — they cover the same destination, same activity, same narrative arc.
That is exactly the "near-duplicate" we want to block.

**At 0.92 in TF-IDF space:** texts need ~95%+ word overlap. Blog titles share only
1-2 unique words, so most similar pairs fall below this cutoff — but they're not
duplicates by word count, they're duplicates by meaning. TF-IDF misses them.

## Tradeoffs

| Option | Pro | Con |
|--------|-----|-----|
| Lower to 0.80 (TF-IDF rec) | More dups blocked in TF-IDF space | Wrong metric; would over-block unique posts in Titan space |
| Keep 0.92 (PRD) | Calibrated for Titan's semantic space | Not empirically verified on real Titan embeddings |
| Raise to 0.95+ | Fewer false positives | More near-duplicates pass through |

Kept 0.92 because: (a) PRD v1.0 Q13 is the authoritative source, (b) no Titan embeddings
to test against, (c) system fails-open (returns None on embed failure) so risk of
over-blocking is low in practice.

   The real Titan-space calibration may suggest adjusting the threshold.

   draft creation (before publish), so the table is blog_drafts in acp_silver_s4 schema.
   The IVFFlat index on content_embedding (lists=10) is already created in migration 041.

3. **Fail-open semantics**: If Bedrock is unavailable, embedding fails and dedup returns
   None — the draft proceeds. This is intentional (blog creation is not blocked by a
   Bedrock outage).

4. **Script path**: `scripts/calibrate_pgvector_threshold.py` — can be run from any
   environment with numpy installed. Takes ~0.3s without DB, ~2-5s with DB connection.
