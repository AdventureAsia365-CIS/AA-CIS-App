## AA-89: B2B Self-Approval HITL

### Decisions
- Auth: tenant JWT (role=`tenant`) is sufficient for B2B approval — no separate `tenant_admin` role in JWT needed. RLS check is done by matching `sub` (tenant_id) in JWT against the run's `tenant_id`.
- `audit_actor_type` enum created in `acp_shared` schema, not `public`. Migration 021 is standalone (no BEGIN/COMMIT) because ALTER TYPE ADD VALUE cannot run inside a transaction.
- Gate endpoints use body `{run_id, notes}` as specified — run_id in body (not path) to keep URL generic per stage.
- GET `/v1/acp/gate/{stage}/run/{run_id}` added to give portal pages a way to load run summary without adding a dependency on stage-specific endpoints.
- `actor_type='tenant_admin'` recorded in details JSONB (not as a typed column) — consistent with existing hitl_gate2 pattern in v1_s3.py which also uses details JSONB.
- admin secret bypass retained for testing.

### Changed
- PRD specifies `actor_type` as a column in audit_log schema — existing migration 030 uses `actor VARCHAR(64)` + `details JSONB`. We store actor_type in details JSONB instead. Migration 021 only creates the enum type; it does not ALTER the existing audit_log table.
- Stage `s4` is supported in the API (forward-compat) even though S4 is not yet implemented — endpoint returns 404 gracefully if no hitl_request row found.

### Tradeoffs
- Single router `v1_acp_gate.py` vs separate per-stage files: single file chosen — all 3 stages share identical logic, just a stage integer mapping difference. Easier to review.
- Portal pages: 3 separate page.tsx files (s2/approve, s3/approve, s4/approve) vs a single dynamic `[stage]/approve` page: separate files chosen — each stage will eventually show stage-specific data. Avoids premature abstraction.

### Should know
- acp_hitl_requests table lives in acp_shared schema (defined in acp_m0_migration.sql applied via ECS, not in numbered migrations). Schema: hitl_id UUID, run_id UUID, stage INT, status TEXT, expires_at TIMESTAMPTZ, reviewer_id TEXT, reviewer_notes TEXT, resolved_at TIMESTAMPTZ.
- Stage mapping: s2→2, s3→3, s4→4 (string from URL path → integer in DB).
- The existing Gate 2 endpoints in v1_s3.py only allow hitl_reviewer role. The new gate endpoints allow tenant role — this is the core B2B change.
