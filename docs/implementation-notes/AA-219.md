# AA-219 — Surface judge/quality/audit/failure/revalidate + full DFS on VersionCompare

[AA-193·UI] Blocked by AA-218 (shipped, migration 069 live).

## Decisions
- Reuse the `/detail` endpoint's judge extraction (`_parse_meta` + `meta.get("judge")`)
  and the `list_tour_versions` jsonb guard idiom (str→json.loads / else list()) — do not
  invent a new helper.
- `revalidate_ran` / `revalidate_passed` read from `gc.metadata` (AA-213 persists them there),
  not a dedicated column.
- DFS (keyword_ideas/PAA/related/seed) is per-tour-seed, so it can look identical across
  versions of the same tour — UI labels it "DFS per tour seed" rather than per-version.

## Changed
- `/versions/{num}` (get_tour_version_detail): SELECT + LATERAL gain the audit/quality/fix/DFS
  columns; return gains 14 fields (judge, score_quality, brand_audit_*, fix_pass_*, failure_codes,
  revalidate_ran/passed, keyword_ideas, people_also_ask, related_keywords, seed).
- Frontend `VersionDetail` type + VersionCompareModal render (quality bar, judge block,
  failure/revalidate badges, DFS section).

## Tradeoffs
- Backend already declared brand_audit_*/fix_pass_* in the frontend type but never returned them
  → silent-empty Brand Audit section. Fixing the contract end-to-end here.

## Should know
- Jsonb columns come back as str OR list/dict depending on driver path — every parse uses the
  guarded idiom from list_tour_versions:1831-1845.
- `/detail` (1950-2006) is the source-of-truth pattern being copied.
