## AA-80: ValidatorAgent Port

### Decisions
- **Input format**: blog as flat `dict` with `content_md` (markdown), not structured `BlogArticle`. Markdown is parsed locally via `_parse_markdown()` to produce sections/FAQ/paragraphs.
- **30+ checks**: 30 structural/content checks (GROUP A/B/C defined below) + 1 aggregated compiled-rules check that iterates all db_rules and creates one CheckResult per firing rule. Total checks scales with db_rules count (currently 32 rules → potential 62 CheckResults).
- **No OpenAI**: LLM tone check from original validator_agent.py skipped. Our stack uses Bedrock only, and the acp-s4-evaluate Lambda already handles LLM scoring in generate.py.
- **LEAK_PHRASES/PROCESS_LANGUAGE_PHRASES**: Copied verbatim from models.py (Ms. Thư). Supplemented with task spec phrases.
- **RAW_DATETIME_PATTERN / RAW_FIELD_BLOB_PATTERN**: Copied from models.py (Ms. Thư).
- **db_rules stage=NULL**: All system rules have stage IS NULL (global). Query uses `WHERE is_active = TRUE AND stage IS NULL` to avoid type cast error (stage is smallint).
- **sample_loader integration**: `_check_notes_style` lazy-loads `get_sample_metrics()` on first call (cached). `sample_loader.compute_style_metrics(sample_text)` returns avg_paragraph_length which drives thin-paragraph threshold.
- **Not-do bad blog**: `blog1_south-korea-soft-adventure-destination.html` used for integration test — stripped HTML to text, verifies ≥5 checks fail (has "operational note" ×many, datetime stamps, "verify provider details", generic opening).

### Changed
- `_check_broad_destination` — only runs when `brief.get("destination","").lower()` contains "korea" or "south korea" (original hardcoded Korea-specific place names). Returns pass for other destinations.
- `_check_h1_matches_topic` — threshold lowered to 0.3 from 0.6 (original) because we match title against primary_keyword which may be a keyword phrase not verbatim heading text.
- `_build_repair_targets` — returns list[dict] not list[str] (our ValidationResult schema requires {section, check_name, hint}).

### Tradeoffs
- Flat markdown parsing vs structured model: markdown parsing is imprecise (FAQ extraction is heuristic). Acceptable for M4 — will improve as S4 LLM output format solidifies in M5.
- db_rule checks as separate CheckResults: makes the checks list long but gives full transparency per rule. RepairAgent (AA-81) needs this granularity.

### Should know
- `ValidatorAgent.__init__` takes `db_rules: list[dict]` — caller must fetch from DB first (generate.py does this).
- `validate()` is async to match generate.py integration — all internal checks are synchronous.
- `_check_compiled_rules()` runs BOTH block and flag rule types against content. Block = fail immediately. Flag = adds to review_flags but doesn't fail the check (unless score_gate). Score_gate not yet in compiled rules seed.
