# AA-220 — Multi-version export (A) + DOCX (B) + frontend (C) + multi-tour filter (D)

## Decisions
- **Endpoint A path** = `GET /tours/{tour_id}/versions/export?versions=4,5,6&format=csv|xlsx`.
  Registered ABOVE `GET /tours/{tour_id}/versions/{version_num}` (admin_pipeline.py) because
  `version_num` is typed `int` — `"export"` 422s on that route instead of falling through.
- **Reused the get_tour_version_detail SELECT** but `version_num = ANY($2::int[])` for N versions.
  `brand_name` comes from `JOIN shared.tenant_brand_rules` (metadata->>'brand_rule_id'); model = `gc.model_editorial`
  (generated_content has NO brand_name/model_tier column — verified STEP 0).
- **Table orientation:** field × version. Column 0 = `field` (row label), one column per version (`v4`, `v5`…).
  XLSX uses `freeze_panes=(1,1)` so header row + field column stay pinned.
- **Score rows:** included all of score_brand / score_seo / score_structure / score_quality / score_overall
  (the scope wrote "score_quality(overall)" — kept both score_quality sub-score AND score_overall headline to avoid data loss).
- **judge dict keys** (from admin_pipeline.py:153-158): brand_fit / distinct / mission_present / feedback / judge_score
  → mapped to judge_fit / judge_distinct / judge_mission / judge_feedback / judge_score rows.
- `_as_list` copied inline (module convention — no shared helper).

# AA-220 display-extend — score detail in history (H1) + compare labels/G2 (H2) + per-version DOCX in history (H3)

## Decisions
- **H1 needed a backend change** (originally scoped frontend-only). The Rewrite History table renders
  `TourVersion[]` from `GET /tours/{tour_id}/versions` (`list_tour_versions`), which returned ONLY
  `score_overall` (as `quality_score`). Sub-scores + judge live on `VersionDetail` (the per-version
  *detail* endpoint), NOT the list. STEP 0 conflated the two interfaces. With user sign-off, extended
  the list endpoint SELECT (the `qs` JOIN was already present) — minimal, no JOIN/WHERE/shape change.
- **judge_score parse** = promoted to module-level `_extract_judge_score(metadata)` so it's unit-testable
  (was a nested closure). Mirrors `get_tour_version_detail`'s canonical str-OR-dict metadata handling.
- **Sub-score null guard** uses `is not None` (keeps a legit `0.0`), unlike the older `quality_score`
  line on the same endpoint which uses truthy — chose correctness over local consistency.
- **H3 helper extraction:** moved `downloadBlob` + `exportDocx` OUT of `VersionCompareModal` to
  page-scope (`downloadBlob` pure, `exportVersionDocx(tourId, vnum)`) so both the modal DOCX button
  and the history row can call one shared helper. `exportComparison` stays in the modal (depends on
  `panels` + `setExporting`) and now `await`-wraps the shared `downloadBlob` to keep its spinner.
- **G2 (H2):** relabeled compare-panel bars "Overall (gate)" / "Quality (validate)" + added optional
  `tooltip` prop to `ScoreBar` (native `title`) + a muted note clarifying Overall=min(validate,judge)
  gate vs Quality=validate sub-score. These two are distinct metrics; users were reading them as one.

## Changed
- `list_tour_versions` SELECT (admin_pipeline.py): +`qs.score_brand/seo/structure/score_quality`, +`gc.metadata`;
  response dict +`score_brand/score_seo/score_structure/score_quality/judge_score`. Existing `quality_score`
  (=score_overall) untouched → no caller breaks.
- New module-level `_extract_judge_score(metadata)`.
- master-content/page.tsx: `TourVersion` interface +5 optional fields; history header `Score`→`Overall` +5 cols
  (Brand/SEO/Struct/Quality/Judge); 5 new row `<td>` with scoreColor tint + "—" fallback; `ScoreBar` tooltip prop;
  compare-panel labels + note; per-version **DOCX** button in history row (View · DOCX · Promote); `downloadBlob`/
  `exportVersionDocx` moved to page scope.

## Tradeoffs
- Modal DOCX button lost its `exporting` spinner toggle (download is instant; `disabled={exporting}` still valid,
  just always-false during DOCX). Comparison export keeps the spinner. Accepted to avoid duplicating downloadBlob.

## Should know
- The new history columns are populated ONLY because the list endpoint was extended — pure-FE would have shown "—".
- judge_score is None for legacy/no-profile versions (judge skipped) → renders "—". Expected, not a bug.
