# AA-305 — decompose_atoms() dual-path routing + poller

## Decisions

- **<100 path: sequential loop, no concurrency.** `invoke_claude()` has no
  built-in timeout/retry, and acc1 InvokeModel RPM/TPM has never been
  measured (grepped whole repo + Terraform, nothing found). Concurrency here
  would be guessing blind. Real pool today is 1 tour, so the latency cost of
  staying sequential is negligible. Revisit once AA-299/AA-300 grow the pool
  and there's real rate-limit data.
- **job_arn column added, not asked for explicitly.** `create_model_invocation_job()`'s
  response was never captured before this PR. Without storing the real
  jobArn, `GetModelInvocationJob` can't be called later — `jobIdentifier`
  must be the full ARN, a bare job name is rejected (verified live against
  atomjob_cf3d9066e0). This is a hard prerequisite for the poller to exist
  at all, not scope creep — added it to migration 082 alongside the
  requested succeeded_count/failed_count columns.
- **succeeded_count/failed_count read from GetModelInvocationJob response
  fields, not manifest.json.out.** Original instruction said manifest.json.out;
  AWS docs (fetched live, cited in code) show the same GetModelInvocationJob
  call already made for status also returns totalRecordCount/
  processedRecordCount/successRecordCount/errorRecordCount directly — the
  docs explicitly say these are the same numbers as the manifest file.
  Reusing the same call avoids a second S3 round trip and extra IAM surface
  for data already in hand. Flagged this deviation before implementing.
- **status='completed' if >=1 tour succeeded (inline path), 'failed' only if
  0 succeeded.** No 'partial' value exists in the status CHECK constraint
  (migration 081) and adding one wasn't in scope. Full per-tour breakdown
  always lives in error_message (JSON) regardless, so a partial run is never
  indistinguishable from a clean one even though status is binary.
- **Poller trigger mechanism deliberately NOT decided/wired.** Investigated
  AA-CIS-Infra/modules/eventbridge (EventBridge Scheduler + custom bus exist,
  no Lambda target wired to it yet) and AA-Bedrock-Invoker's trust policy
  (currently scoped only to aa-cis-dev-ecs-task-role — a new Lambda role
  would need adding). Two live options: (a) EventBridge Scheduler + new
  Lambda (needs IaC + trust policy change), (b) on-demand admin endpoint
  calling poll_job() (no new AWS infra, but needs ECS running when called,
  which conflicts with cost-discipline auto-stop). `poll_job()` /
  `list_pollable_job_ids()` were written trigger-agnostic specifically so
  this decision doesn't block the rest of the PR.

## Changed

- `api/routers/v1_atoms.py`: threshold branch (`len(rows) < 100`) added
  right after tour selection, before the existing Batch-submit code (which
  is otherwise untouched — same request-building logic). New
  `_decompose_inline()` function, new `_strip_json_fence()` helper. Batch
  submit path now captures and stores `create_resp["jobArn"]`.
- `api/migrations/082_atom_decompose_jobs_dual_path.sql` (new, applied live
  to dev DB): `+job_arn TEXT`, `+succeeded_count INTEGER`,
  `+failed_count INTEGER` on `acp_contract.atom_decompose_jobs`. Additive
  only, no constraint change, no backfill for pre-existing rows (except one
  manual one-off backfill of `job_arn` on `atomjob_cf3d9066e0` purely to
  make that historical row pollable for the live test — not a code change).
- `shared/services/atom_decompose_poller.py` (new): `poll_job()`,
  `list_pollable_job_ids()`. Not imported/called from anywhere yet — no
  trigger wired.
- `~/.claude/CLAUDE.md` (global, outside this repo): CIS DEV account table
  entry and all example commands corrected from
  `867490540162`/`pqnghiep-admin` to `005097885195`/`aa365-admin`. Old
  account/profile kept in the table, relabeled as the Bedrock satellite
  account (AA-296) rather than removed — still genuinely in use.

- **succeeded_count/failed_count gated on `processedRecordCount > 0`, not on
  key-presence alone.** Found via the real poller test itself:
  `atomjob_cf3d9066e0` (failed pre-flight, never reached `InProgress`) came
  back with `successRecordCount`/`errorRecordCount` PRESENT and reading `0`
  — the original presence-only check wrote a confident-looking `0/0` for a
  job that never processed a single record. AWS docs: "The counters return
  0 when you submit a job but processing has not yet started" — presence
  doesn't mean measured. Since `processed = success + error` always,
  `processedRecordCount == 0` implies both are unmeasured placeholders;
  gating on it costs nothing for jobs that genuinely processed records.
  Re-verified live: same job now correctly returns NULL/NULL.

## Tradeoffs

- Testing methodology: ECS was still running the pre-PR image (this PR
  hasn't merged/deployed), so the actual HTTP endpoint couldn't be
  exercised. Instead, the literal PR source files were copied into a
  non-destructive scratch directory inside the real ECS container (same
  task role, same satellite AssumeRole chain, same dev DB) and invoked
  directly as Python functions, bypassing only the FastAPI routing layer.
  This is real Bedrock calls + real DB writes, not mocked — but it is not
  literally "hit the deployed endpoint over HTTP."
- `_decompose_inline()`'s real test surfaced a genuine bug (model wraps JSON
  in a ```json fence) — fixed in this PR. It also surfaced a second,
  pre-existing bug in `_build_user_prompt()` (HIGHLIGHTS rendering garbled
  for at least one tour whose `aa_highlights` appears double-JSON-encoded)
  that affects both paths equally — NOT fixed here, flagged as a follow-up,
  since it predates AA-305 and fixing shared prompt-building code was not
  in scope.
- Poller's succeeded_count/failed_count mechanism is verified end-to-end for
  real, but only for the "processedRecordCount == 0 -> NULL" branch (the
  only real job available failed pre-flight, 0 records ever processed). The
  opposite branch — "processedRecordCount > 0 -> gate passes, real numbers
  get written" — has NOT been observed. Can't be, until AA-299/AA-300 grow
  the tour pool past 100 and a real job actually reaches InProgress.
