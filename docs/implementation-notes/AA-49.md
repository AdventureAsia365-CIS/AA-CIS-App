## AA-49: Harness Engineering Layer H-1 + H-2 (M4)

### Decisions
- **Migration 032 already applied**: blog_drafts and acp_hitl_requests already had all 6 new columns in live DB when session started. Created file as idempotent (IF NOT EXISTS) for repo history completeness. Did NOT re-apply.
- **acp_output_rules schema differs from spec**: Actual columns are `rule_type`, `pattern`, `action_value`, `is_active` — not `rule_code`/`action_type`/`condition_field`/`condition_op`/`condition_value`/`priority`. Post-processor adapted to actual schema. Rules are text-pattern based (pattern = text substring to match).
- **Lambda path**: `services/acp_s4_evaluate/handler.py` — consistent with existing Lambda structure (all Lambdas live in `services/`).
- **S4 flow stub**: No existing S4 service found. Created `services/acp_s4/` as new service following acp_s3 pattern, with generate.py showing integration points. S4 full implementation is M5+.
- **MODEL_ID**: `us.anthropic.claude-haiku-4-5-20251001-v1:0` (inference profile per CLAUDE.md primary model).
- **post_processor logic**: Since `pattern` is a plain text substring (not regex, not field-specific), apply_output_rules checks `pattern in content.lower()` for all rule types. Field-specific logic (word_count, seo_title length) not in DB schema — skipped.
- **api/services/ location**: Created `api/services/acp_post_processor.py` per task spec. This directory is new (api/ previously only had routers/middleware/lambda_authorizer).

### Changed
- `apply_output_rules()`: Removed condition_field/op/value parameters (don't exist in DB). Pattern matching is purely text-substring based on `acp_output_rules.pattern`.
- Lambda Terraform: Added `acp_s4_evaluate_zip` variable and resource following acp_s3_campaign_planner pattern. No VPC attachment since Lambda doesn't call DB directly.
- Rules API: Uses actual column names (is_active, rule_type, pattern) instead of spec's active/action_type/condition_*.

### Tradeoffs
- **No VPC for evaluate Lambda**: Lambda only calls Bedrock (public endpoint), no DB. Avoids cold start penalty of VPC Lambda (~500ms) and simplifies IAM. If DB access needed later, add VPC config.
- **Text-pattern post-processor**: Current rules are brand-voice word replacements (cheap→exceptional value, book now→blocked). Not regex-based. More complex condition logic deferred to when rule schema is extended.
- **S4 stub vs full implementation**: Creating a generate.py stub that shows integration points is enough for H-2 review. Full S4 blog generation is M5 scope.

### Should know
- acp_output_rules.stage is NULL for all system rules (SYS-001→010) — meaning they apply to all stages. The WHERE clause `(stage = $1 OR stage IS NULL)` is correct.
- acp_output_rules.tenant_id is NULL for system rules — `(tenant_id = $1 OR tenant_id IS NULL)` handles both system-wide and tenant-specific rules.
- The evaluator Lambda must receive ONLY content text (no brand context, no outline) — this is the isolation guarantee for H-1. Do not add additional fields to the payload.
- evaluator_input_hash = SHA256 of the raw blog text — proves what the evaluator saw, audit trail.
