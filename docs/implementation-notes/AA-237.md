# AA-237 — Make haiku→sonnet auto-upgrade opt-in (default OFF)

## Decisions
- New field `allow_auto_upgrade: bool = False` added to `TourRunRequest` only (the model
  `_execute_run_tour(req)` receives). `IngestS3Request` also carries `model_tier`/`subtitle_focus`
  but does NOT feed `_execute_run_tour` directly — it constructs its own `TourRunRequest` per tour
  internally — so the flag is not added there. Net effect: batch S3 ingest never auto-upgrades
  unless that constructor is later updated to pass the flag through (acceptable: default OFF = the
  whole point of AA-237).
- Upgrade gate (L426-430) gains `and req.allow_auto_upgrade` as the final AND term so the Sonnet
  re-run only fires when the caller explicitly opts in.
- Response surfaces `auto_upgraded: bool`. `model_used` / `cost_usd` already reflect the upgraded
  run (because `result` is reassigned to `_upgraded`), so no separate "both models/cost" keys are
  added — existing keys carry the upgraded values and the new bool says whether that happened.

## Changed
- Previously: every haiku run scoring 0 < score < AUTO_UPGRADE_THRESHOLD (8.5) silently re-ran on
  Sonnet and got charged Sonnet cost.
- Now: that block is gated behind `req.allow_auto_upgrade` (default False) → no silent Sonnet charge.

## Tradeoffs
- Could have extracted a pure predicate function for the gate; kept the inline `if` and instead
  unit-tested `_execute_run_tour` with `_rewrite_tour` mocked + DB/SEO/export stubbed, asserting the
  Sonnet branch call-count (0 vs 1). This tests the real wired condition, not a copy.

## Should know
- `auto_upgraded` is computed from a local flag set inside the gated block, not by comparing models
  after the fact (a caller could legitimately request `model_tier="sonnet"` with no upgrade).
- The upgrade still only *keeps* the Sonnet result if it scores higher than haiku (existing
  behavior preserved).
