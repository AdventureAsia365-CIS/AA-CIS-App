# AA-209 — Judge scoring reliability + surface judge to UI

Branch: `feature/aa-209-judge-reliability-surface` (base `develop @ 08dae3e`).
Build order: PART 1 (client determinism) → PART 2 (persist judge to metadata) → PART 3 (surface to UI).

## Root cause (verified in diagnostic, not re-investigated)
`quality_scores.score_overall` is persisted from `result["quality_score"]` (`admin_pipeline.py:378`),
which `judge_node.py` overwrote to `min(validate_score, judge_score)`. `judge_score` is a GPT-4.1
output, but `shared/llm_client/client.py:_call_openai` never forwarded `temperature`/`seed`, so the
judge ran at OpenAI default temperature (1.0) with no seed → non-deterministic → same content gave
different `score_overall` across versions (probe: v4=7.0 vs v5=9.0 on identical sub-scores). The 4
sub-score columns come from validate's `sub_scores` (judge-independent), and the UI `/detail`
endpoint returns only the 4 legacy + overall, so Overall looks decoupled from its components.

## PART 1 — judge reliability (client layer) — DONE

### Decisions
- **Forward-if-present via `request.model_fields_set`** (not a value check) in `_call_openai`.
  `temperature` always has a value (default 0.7), so "is it set?" can only be answered by pydantic's
  `model_fields_set`. This guarantees the T3 content-fallback path — `graph.py:199` /
  `flag_fix_node.py:144`, which never pass `temperature`/`seed` — is byte-for-byte unchanged.
- **`seed` added to `LLMRequest`** as `Optional[int] = None` (no migration; it's a request model).
- **Fixed judge seed = 42, temperature = 0.1**, exposed as module constants `_JUDGE_SEED` /
  `_JUDGE_TEMPERATURE` in `judge_node.py` so the test can assert against them.

### Changed
- `shared/llm_client/models.py`: added `seed: Optional[int] = None` to `LLMRequest`.
- `shared/llm_client/client.py` `_call_openai`: build `kwargs` dict; forward `temperature` when in
  `model_fields_set`, forward `seed` when in `model_fields_set` and not None.
- `services/content_generation/judge_node.py`: request now passes `temperature=_JUDGE_TEMPERATURE`
  and `seed=_JUDGE_SEED`.
- `tests/unit/test_aa209_judge_determinism.py`: 5 tests (forward-when-present, omit-when-absent,
  independent temperature-only forwarding, judge builds reproducible request, seed defaults unset).

### Tradeoffs
- `model_fields_set` is the only way to distinguish "caller set 0.7" from "default 0.7". A simpler
  `temperature is not None` check would have changed T3-fallback behavior (it would start forwarding
  0.7), so it was rejected.
- Seed 42 is arbitrary-but-fixed. OpenAI seeds are best-effort, not a hard determinism guarantee,
  but combined with temperature 0.1 it removes the large run-to-run variance that caused the bug.

### Should know
- CI flake8 ignores E221/E302 (`.github/workflows/ci.yml` extend-ignore); a bare `flake8` will
  report false positives on pre-existing aligned `=`. Lint with the CI flags.
- Existing `test_aa206_judge.py` (7 tests) still green after the judge_node change.

### Tests

## PART 2 — persist judge scores to metadata (no migration) — DONE

### Decisions
- **Extracted `_build_generated_metadata(...)`** (module-level in `admin_pipeline.py`) instead of
  inlining the judge merge. The generated_content write is a *new-version INSERT*, so there is no
  prior DB metadata to read — "existing keys" means the base keys assembled in code. A pure helper
  makes the merge unit-testable without standing up the endpoint + asyncpg pool + brand resolution.
- **Guard = `result.get("judge_brand_fit") is not None`.** judge_node omits all `judge_*` keys when
  the judge skips/fails, so absence → no `metadata.judge`. Used `is not None` (not truthiness) so a
  legitimate `brand_fit`/`judge_score` of `0.0` and `mission_present=False` still persist.
- **`metadata.judge` shape:** `{brand_fit, distinct, mission_present, feedback, judge_score}`. Maps
  judge_node's `judge_cross_brand_distinct` → `distinct` for a compact UI-facing key.
- **`json.dumps(..., default=str)`** on the metadata write (Decimal-safe), replacing the bare dumps.

### Changed
- `services/content_generation/judge_node.py`: return dict now includes `"judge_score": judge_score`
  (the capped value that is min()'d against validate — the actual driver of score_overall).
- `api/routers/admin_pipeline.py`: added `_build_generated_metadata`; the version-INSERT path now
  builds `metadata_val` via that helper with `default=str`.
- `tests/unit/test_aa209_judge_persist.py`: 6 tests (judge populated, base keys preserved, judge
  absent when not run, falsey-score guard, Decimal-safe JSON, judge_node returns judge_score).

### Tradeoffs
- A helper adds one indirection but is the only way to test the merge without a DB. The base-key
  dict is byte-for-byte the original (same keys/values/order), so no behavior change for non-judge
  rows beyond the added `default=str`.
- Pre-existing `datetime.datetime.utcnow()` deprecation warning was carried into the helper verbatim
  (not fixed here — out of scope, would be an unrelated change).

### Should know
- No migration: judge data rides in the existing `generated_content.metadata` JSONB. PART 3 reads it
  back via `metadata->'judge'` in `get_tour_detail`.
- Regression: AA-206 (7) + PART 1 (5) + PART 2 (6) = 18 passed. CI flake8 clean.

## PART 3 — surface judge to UI (endpoint + frontend) — DONE

### Decisions
- **CompareModal compares multiple TOURS, not versions.** Confirmed `CompareModal({ tourIds })`
  maps each id to its own `/detail` fetch (latest version). So keeping `get_tour_detail` at
  `ORDER BY version_num DESC LIMIT 1` breaks nothing — there was never a multi-version fetch.
- **Judge-capped badge rule:** show when `score_overall < validate_avg − 0.01`, where validate_avg =
  mean of the 4 sub-scores (brand/seo/structure/quality). The 0.01 epsilon avoids float-noise false
  positives. Only computed when all 4 sub-scores are present (else no badge).
- **Graceful judge guard:** `metadata.judge` may be absent (legacy/no-profile versions) → backend
  returns `judge: null`; both components render "not run" + "—", never crash. Same for null
  `score_quality` / `brand_audit_status`.
- **Backend metadata parse is defensive:** `_parse_meta` json.loads a stringified jsonb (asyncpg's
  default) and falls back to `{}` on malformed JSON, so a corrupt metadata row can't 500 the detail
  view (covered by test_detail_no_crash_on_malformed_metadata).

### Changed
- `api/routers/admin_pipeline.py` `get_tour_detail`: SELECT now also pulls `gc.metadata` +
  `qs.brand_audit_status`; response `generated` gains `score_quality`, `brand_audit_status`, and
  `judge` (parsed from `metadata.judge`).
- `frontend/.../CompareModal.tsx`: added `JudgeMeta` type + 3 fields; `fmtScore`/`validateAvg`
  helpers; Quality ScoreRow; judge-capped badge next to Overall; compact Brand Judge section
  (fit/distinct/mission ✓✗ + brand_audit_status badge + feedback prose), all guarded.
- `frontend/.../TourDetailPanelV2.tsx`: added `JudgeMeta` + 2 fields to `TourDetailFull.generated`;
  enriched the rewrite-tab info bar with the 5-score breakdown, judge-capped flag, and the same
  Brand Judge block.
- `tests/unit/test_aa209_detail_endpoint.py`: 4 async tests (judge+quality present, judge None when
  absent, jsonb-string parsed, malformed-metadata no-crash) using a fake pool/conn.

### Tradeoffs
- Backend judge object is returned as-is with the PART 2 key name `distinct` (not
  `cross_brand_distinct`); the frontend reads `judge.distinct` to match. Single source of truth.
- TourDetailPanelV2 uses an inline IIFE for the breakdown rather than a shared component — local,
  one-use, avoids a new export; matches the file's existing inline-render style.
- `.next/dev/types/validator.ts` had stale `(admin)` route-group refs (group was renamed to
  `admin/` in Session 31). Not my code — a fresh `npm run build` regenerates them clean.

### Should know

  errors from the stale `.next` cache; the real build is clean.
