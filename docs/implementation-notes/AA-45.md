## AA-45: S3 Campaign Planner Lambda + Gate 2 HITL

### Decisions
- Migration 031 adds 3 extra tables beyond spec's `ads_plan`: `acp_run_context`, `acp_lessons_agency`, `acp_lessons_shared` — all required by Lambda Step 1/7/8 but not yet defined anywhere in migrations
- Dropped invalid FK `REFERENCES acp_shared.tenants(tenant_id)` on `ads_plan.tenant_id` — `acp_shared.tenants` does not exist; following M0 pattern of VARCHAR(50) without FK
- `acp_hitl_requests` INSERT uses actual M0 column names (`stage`, `gate_type`, `reviewer_id`) not spec pseudo-names (`gate`, `reviewer_email`) — confirmed from DBeaver/acp_m0_migration.sql
- Lambda lives at `services/acp_s3/` (not `services/acp/s3/`) to match `services/acp_brand_brief_parser/` sibling pattern
- API routes registered under prefix `/acp/s3` in main.py to match `/acp/s1`, `/acp/s2`
- HITL Gate 2 routes added to new `api/routers/v1_s3.py`, not to v1_hitl (no v1_hitl router exists)
- `acp_run_context` uses one row per run_id with nullable JSONB columns per stage — S3 Lambda UPDATEs its 3 columns in Step 9

### Changed
- Spec says `gate` column in hitl INSERT — actual M0 schema uses `stage` (SMALLINT) and `gate_type` (VARCHAR)
- Spec says `reviewer_email` — actual column is `reviewer_id`
- content_calendars table already exists from M0 with different column set: `strategy`, `calendar_weeks`, `ad_copy` vs spec's `expanded_markdown` — Lambda will store expanded calendar in `calendar_weeks` JSONB

### Tradeoffs
- Put all 4 new tables in single migration 031 vs separate files: single file is simpler and avoids ordering issues for tables that FK to each other
- Storing `expanded_markdown` as-is in content_calendars.calendar_weeks (JSONB text field) vs new column: chose to add `expanded_markdown TEXT` column to content_calendars via migration 031 to preserve type fidelity
