# AA-250 B2 — S1 pipeline stage-progress bar

## Problem
The S1 async job (`POST /admin/run-tour-async` + `GET /admin/jobs/{id}`, AA-223/AA-248)
only had 2 status writes — `mark_running` at dispatch, `mark_succeeded`/`mark_failed` at
the end. The LangGraph run in between (`generate → validate → llm_judge → [retry loop] →
brand_audit → flag_fix → revalidate`, ~2.5–3.5min/tour per live sample, STEP 0) had no
persisted sub-stage, so the FE only showed a static "Processing…" spinner for the whole
duration with no indication of progress.

## Decisions
- **Migration 076**: `shared.pipeline_jobs.current_stage TEXT`, nullable, plain
  `ADD COLUMN IF NOT EXISTS` (no `ALTER TYPE`, no transactional-safety caveat). Applied
  Dev via S3-mediated ECS exec, verified column exists + `schema_versions` row inserted.
- **`jobs_repo.update_stage(job_id, stage)`**: writes `current_stage` AND `heartbeat_at`
  together — reuses the same column `sweep_interrupted()` already reads for liveness, so a
  job actively streaming stage updates counts as alive with zero new liveness plumbing.
- **Graph streaming, not a full architecture change**: `_rewrite_tour` (`v1_pipeline.py`)
  switched from `graph.invoke()` (wrapped in `run_in_executor` to avoid blocking the event
  loop) to `graph.astream(initial_state, stream_mode="updates")`, driven directly as a
  native async generator. Confirmed via `langgraph/_internal/_runnable.py` that sync node
  callables are auto-dispatched to a thread-pool executor internally when a graph runs in
  async mode — the old manual `run_in_executor` wrapper is no longer needed, event loop
  stays unblocked without it.
- **Final-state reconstruction without a checkpointer**: `build_graph()` is compiled with
  no checkpointer (confirmed by reading `graph.py`), so `aget_state()` isn't available.
  Instead: every node function in `graph.py` (`generate_node`, `validate_node`,
  `judge_node`, `brand_audit_node`, `flag_fix_node`, `increment_retry`, `revalidate_node`)
  returns `{**state, ...}` — verified by grepping every `return {` in all 4 node files.
  Since `stream_mode="updates"` emits `{node_name: node_output}` per step and `node_output`
  is always the *complete* state (not a partial delta), tracking "most recent node_output
  seen" reconstructs exactly what `graph.invoke()` used to return — no manual state merging
  logic needed, just `result = node_output` on every event.
- **`on_stage` is opt-in via `job_id`**, not a required param threaded everywhere:
  `_execute_run_tour(req, job_id=None)` builds the callback only when `job_id` is given.
  `_run_tour_job` (async job path) passes its `job_id`; the sync `/admin/run-tour` endpoint
  and the batch `/v1/pipeline/run` upload path (which calls `_rewrite_tour` directly with no
  job row) both pass nothing — behavior for those is byte-for-byte unchanged, `on_stage`
  stays `None`, no `update_stage` calls happen.
- **FE stage→step collapse**: 7 real node names collapse to 4 user-facing steps (`generate`
  →1, `validate`/`llm_judge`/`increment_retry`→2, `brand_audit`/`flag_fix`→3, `revalidate`
  →4) because the 3 middle nodes are perceived by the user as one "scoring" phase, and the
  2 audit nodes as one "brand audit" phase.
- **Retry detection without `retry_count`**: backend doesn't expose `retry_count` on the
  job poll response (only `current_stage`). FE infers a retry loop by counting *distinct
  entries* into the `generate` stage (a ref-tracked counter incremented only on a real
  transition into `generate`, not on repeated polls hitting the same node) — count > 1 means
  the graph looped back via `increment_retry`. When that happens, the stepper does not
  regress to step 1 on re-entering `generate` — it stays pinned at step 2 with a small
  "retry" badge, since the user is still conceptually in the validate-retry loop.

## Tradeoffs
- `current_stage` is last-node-*completed*, not last-node-*started* — during a single node's
  execution (e.g. the ~seconds-long Bedrock call inside `generate`) the FE still just shows
  whichever stage completed most recently, no "in-progress within a stage" sub-indicator.
  Accepted: the 5s poll interval (AA-248) already caps granularity there.
- Retry-loop detection is a heuristic (count of `generate` re-entries seen across 5s polls),
  not the backend's authoritative `retry_count`. A poll landing exactly between two very fast
  retries could in theory undercount — accepted per spec ("Không cần đếm chính xác retry_count").
