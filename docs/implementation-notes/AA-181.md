## AA-181: ACP tenant-auth — collapse to single-header (raw X-API-Key) end-to-end

### Decisions
- `verify_tenant_api_key` is multi-credential (per user clarification, not in original spec):
  accepts EITHER `X-Admin-Secret` (matches `ADMIN_SECRET` env var → AA internal sentinel,
  `reviewer_type=aa_internal`, `sub=tenant_id=AA_INTERNAL_ADMIN_SUB`) OR `X-API-Key` (raw `cis_...`
  key, sha256-hashed and matched against `shared.tenants.api_key_hash` →
  `reviewer_type=tenant_self`). 401 if neither matches.
- Single dependency function lives in `api/routers/auth.py` and is imported under
  router-local aliases (`_get_tenant`, `_get_admin`, `_get_tenant_actor`) so all existing
  `Depends(...)` call sites and `tenant.get("sub"/"role"/"actor")` usages needed zero changes
  beyond the import/definition swap.
- `AA_INTERNAL_ADMIN_SUB = "00000000-0000-0000-0000-000000000001"` preserved from the old
  admin-secret bypass (AA-22 sentinel) so downstream tenant_id comparisons keep working.
- Did NOT use the `acpcore.errors.ACPError`/AA-58 envelope for the 401 — no AUTH-specific
  error code exists in that contract and no exception handler is registered in `main.py`
  for it; plain `HTTPException(401, ...)` matches the existing pattern in this file
  (`tenant_login`, `verify_jwt`).
- `/auth/tenant-login` kept, marked `deprecated=True` + docstring note pointing to
  `verify_tenant_api_key`. Not deleted (removal is a follow-up per spec).
- New `_audit_actor_type()` helper in `v1_acp_gate.py` maps `reviewer_type` →
  `acp_shared.audit_actor_type` enum (`aa_internal` → `hitl_reviewer`, `tenant_self` →
  `tenant_admin`) — reuses existing enum values from migration 021, no new migration needed.
  This mapping is a placeholder for AA-186 (full reviewer_type audit) and may need a
  dedicated enum value later if `tenant_self` audit entries need to be distinguished from
  the old hardcoded `tenant_admin` writes.

### Changed
- 4 routers (`v1_acp.py`, `v1_s1.py`, `services/acp/s2/router.py`, `v1_acp_gate.py`) had
  their local `_get_tenant`/`_get_tenant_actor` (HTTPBearer+JWT) dependency removed entirely
  and replaced by an aliased import of `verify_tenant_api_key`.
- 2 S4 routers (`v1_s4_social.py`, `v1_s4_blog.py`) had their local `_get_admin`
  (X-Admin-Secret-only) dependency removed and replaced the same way — these routers now
  ALSO accept tenant `X-API-Key` (additive, not a behavior removal).
- `v1_acp_gate.py`: renamed `jwt_tenant_id` → `caller_tenant_id` (no JWT involved anymore);
  `gate_approve`/`gate_reject` audit_log INSERTs now write `_audit_actor_type(tenant)`
  instead of the hardcoded `'tenant_admin'::acp_shared.audit_actor_type`.

### Docs/OpenAPI (STEP 4)
- Added two `APIKeyHeader` security schemes (`TenantApiKey` → `X-API-Key`, `AdminSecret` →
  `X-Admin-Secret`) as `auto_error=False` `Security(...)` params on `verify_tenant_api_key`
  in `api/routers/auth.py`. FastAPI propagates these to every route that depends on it, so
  Swagger UI now shows both headers + an "Authorize" lock for all 6 touched routers, without
  changing runtime auth logic (still reads raw `request.headers` for the actual lookup).
- Added a one-line "Auth: Single-header tenant auth (AA-181) — X-API-Key (tenant) or
  X-Admin-Secret (AA internal)." line to the module docstring (→ OpenAPI tag/description)
  of all 6 touched routers (`v1_acp.py`, `v1_s1.py`, `services/acp/s2/router.py`,
  `v1_acp_gate.py` already had one, `v1_s4_blog.py`, `v1_s4_social.py`).
- "ACP wiki canonical auth flow" from the original spec: no ACP wiki exists in this repo
  (GitHub wiki disabled, no `docs/` page for ACP auth) — the OpenAPI docstrings above are
  the canonical in-repo source for now. Flagging for follow-up if a separate wiki/Notion
  page needs updating; out of scope to create new doc infrastructure for this PR.

### Tradeoffs
- Kept the admin-secret path even though Option A says "single header" — the user's
  clarification explicitly requires multi-credential support (admin secret is a distinct,
  pre-existing internal-staff path that S4 routers already depended on). "Single header"
  means *one* header is sufficient per request, not that only `X-API-Key` is accepted.
- `_audit_actor_type()` is a 2-way mapping (aa_internal/tenant_self → hitl_reviewer/tenant_admin)
  rather than adding a 3rd enum value — avoids a migration in this PR at the cost of losing
  the distinction between "AA staff acting as reviewer" and "tenant self-approving" in the
  enum itself (both still distinguishable via `reviewer_type` in the hitl_requests row /
  JSONB details if needed later).

### E2E verify (STEP 6)
- `GET /v1/acp/runs?limit=1` with `X-API-Key: cis_u1k...` → `200` with run data — full
  path (APIGW custom authorizer + new `verify_tenant_api_key`) works end-to-end with a
  single header.
- Same endpoint with no header → `401 {"message":"Unauthorized"}` (APIGW authorizer).
- Same endpoint with an invalid `X-API-Key` → `403` (APIGW authorizer explicit deny).
- `POST /acp/s1/run` (the actual S1 run-trigger path — note: NOT `/v1/acp/s1/run`, that
  path doesn't exist as a route) with valid `X-API-Key` → `422` (missing `tour_ids`/
  `run_config` in body) — i.e. auth passed, request reached the handler/validation layer.
- The original spec's curl target `/v1/acp/s1/run` returns `404 {"detail":"Not Found"}`
  (FastAPI's own 404, confirming the request reached the app past the authorizer) — also
  satisfies the spec's literal PASS criteria ("gets PAST 401/403"), but `/acp/s1/run`
  above is the real endpoint and was used for the more meaningful check.
