# AA-203 — DFS keyword_ideas persist fix + API + admin UI

Branch: `feature/aa-203-dfs-keyword-ideas-ui` (from develop)
Started: 2026-06-18

## Goal
1. Fix key mismatch so `seo_context.keyword_ideas` stores the AA-197 full-metric list (volume/competition/cpc), not the empty `search_volumes` dict.
2. API: return DFS data for a single tour.
3. UI: admin keyword-ideas table (sortable volume desc) + seed + buyer market.

Constraints (from issue): NO migration · NO cache_key change · NO created_at bump.

## Trace facts (from D1 + runtime D2)
- Client `dataforseo_client.fetch_all()` returns `keyword_ideas` as a **top-level list** of dicts `{keyword, search_volume, competition, competition_index, cpc}`.
- `handler.py:101` persisted `seo_data["keywords"]["search_volumes"]` (a `{kw: vol}` dict, often `{}`) into the `keyword_ideas` column → real ideas dropped.
- `top_keywords` persist (`keywords.top_keywords`) is correct — leave untouched.
- Upsert keyed on `cache_key = make_key(seed, location_code)` → all tours of a (country, market) share ONE row; `ON CONFLICT DO UPDATE` never touches `created_at` (explains "newest row 29/05" despite June upserts).

## Decisions
- **Bug 1 fix:** `handler.py:101` now persists `seo_data.get("keyword_ideas", [])` (top-level list) with `json.dumps(..., default=str)`. `top_keywords` left as `keywords.top_keywords` (already correct). No repo/SQL change — column already stores whatever string `keyword_ideas` holds.
- **Bug 2 API:** `GET /admin/tours/{tour_id}/seo-context` in `admin_pipeline.py` (after `/source`). Decision (A): match stored `keyword_search ILIKE '%country%'` for the tour's `tenant_id`, ORDER BY fetched_at DESC LIMIT 1. No seed re-build. Always 200 — `has_data:false` + empty `keyword_ideas` when no row (never 404 except tour itself missing). jsonb columns parsed via `_as_list`. `buyer_market` derived from `TenantConfigService.get_seo_config().target_market` → `resolve_buyer_market` (try/except → null). `people_also_ask`/`related_keywords` always `[]` (schema has no such columns) + `notes` says so.
- Route placed before `/tours/{tour_id}/versions/...` — distinct literal segment, no FastAPI greedy-match conflict.
- **Bug 3 UI:** Added a **"SEO Keywords" tab** to `TourDetailPanelV2.tsx` (slide-over opened from s1-rewrite tour list) rather than a separate page — keeps DFS data next to the rewrite it informs. New `SeoKeywordsPanel({tourId})` sub-component fetches `/api/admin/tours/{id}/seo-context` via BFF catch-all (proxy injects X-Admin-Secret). Header shows seed + buyer_market (+ location_code) + fetched_at + idea count. Sortable table Keyword|Volume|Competition|CPC, default **volume desc**; competition shows `competition_index ?? competition`; volume `toLocaleString`, CPC `$x.xx`. Empty state when `has_data:false`. PAA/related not rendered (schema doesn't persist them). Reused `adminUi` tokens, no new libs.

## Tradeoffs
- Endpoint matches the stored row by `keyword_search ILIKE '%country%'` (Decision A) rather than the tour's own `tour_id` — because the cache_key collapse means a row's `tour_id` is the last tour that upserted it, not necessarily the one being viewed. Country-match returns the seed the tour actually shares.
- UI table reuses the history-tab inline table style instead of a new shared `<DataTable>` — smaller diff, consistent look, no new abstraction.
- Tests target the persist payload via mocks (no live DB) — fast + deterministic, matches the AA-197 test approach.
