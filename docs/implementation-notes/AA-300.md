# AA-300 — Atom Curation UI (N3)

## Decisions

- **Router**: new `api/routers/admin_atoms.py`, prefix `/admin`, imports `verify_admin_secret`
  from `admin.py` (does not redefine it) — following the exact established convention
  `admin_pipeline.py` itself documents in its own file header ("Auth: x-admin-secret header
  only (no tenant JWT accepted)"). Registered in `api/main.py` the same way as the other 4
  admin router files (`admin`, `admin_pipeline`, `admin_acp_proxy`, `admin_settings`).
- **Auth**: X-Admin-Secret-only, not the dual X-Admin-Secret-or-JWT pattern `v1_atoms.py` uses.
  The existing frontend BFF proxy (`frontend/app/api/admin/[...path]/route.ts`) only ever sends
  `X-Admin-Secret` — confirmed by reading it — never a JWT, so the dual pattern would be dead
  code here. `POST /v1/atoms/decompose` (the one endpoint using dual auth) has zero callers
  anywhere in the repo (grep-confirmed) — not a load-bearing precedent to imitate for an
  internal CIS-admin tool.
- **Proxy**: reused the existing `/api/admin/[...path]/route.ts` as-is — it is a generic
  catch-all that already forwards any `/api/admin/*` path to `/admin/*` on the backend. No
  second proxy route created.
- **`v1_atoms.py` untouched**: confirmed by a dedicated regression test
  (`TestNoV1AtomsRegression`) — this issue is a purely additive, separate resource.
- **No new `tour_atoms` migration**: `distinctiveness`, `visual_potential`, `starred`,
  `deleted`, `media` all already exist (migration 079) and `starred`/`deleted` are already
  consumed by AA-301's allocator (`services/acp_planning/allocator.py`) — confirmed live in
  STEP 0 recon before writing any code.
- **"unreviewed" filter**: derived from `updated_at == created_at` rather than a new column.
  `tour_atoms` has no `reviewed`/`reviewed_at` field, and both timestamps are set to the same
  `now()` in the same INSERT statement (`v1_atoms.py`, migrations 079/084/085) — Postgres
  resolves multiple `now()` calls in one statement to the same transaction timestamp, so they
  stay exactly equal until the first `PATCH` touches the row (which explicitly sets
  `updated_at = now()`). Self-chosen, not specified by the issue text.
- **"thin tour" filter**: reuses the same `tour_atom_count` computed via a JOIN subquery
  (count of non-deleted, non-empty-marker atoms per `tour_id`), compared against
  `THIN_TRIP_ATOM_MIN` (imported from `services.acp_shared.atom_constants` — the same constant
  N5's B5 fix already uses, not redefined).
- **Soft delete = existing `deleted` column**, not `is_empty_marker` (that marks "this
  decompose run legitimately produced zero atoms," a structurally different fact — confirmed
  by `v1_atoms.py`'s own comment on the distinction). "Star" = existing `starred` column.
  Neither needed a migration.
- **Total-count query added to `GET /admin/atoms`**: the existing `Pagination.tsx` component
  (reused as required) needs a total matching-filter item count, not just the current page's
  row count, to compute page numbers — added a second `COUNT(*)` query using the same `WHERE`
  clause as the paginated `SELECT`. Not specified by the issue text; a direct consequence of
  reusing `Pagination.tsx` as instructed rather than building a new pagination UI.
- **Keyboard shortcuts (X = delete, S = star) bind to the currently-hovered card**, not a
  separately-selected/focused one — no extra click needed to "select" a card first, matching
  the issue's own "mỗi giây đều đáng" (every second counts) framing for a 200-tour curation
  session. Ignored while a text input/textarea is focused (editing an atom's text) so typing
  "x"/"s" doesn't fire a shortcut by accident.
- **Preview endpoint hardcodes `markets=["US"], channels=["blog"],
  capacity_posts_per_week=4`** and auto-approves the quarter plan as `"admin-preview-demo"` —
  the same tenant-config gap AA-301's own implementation notes already flagged (no tenant
  campaign-config table exists yet). Auto-approval here does NOT weaken Gate B for the real
  N5/N6 production path — `approve_quarter_plan()` still must be called explicitly by
  something, and this demo endpoint is read-only relative to the database (no INSERT/UPDATE
  anywhere in `runway_map()`/`plan_quarter()`/`allocate_month()`), so nothing persists from a
  demo "approval."
- **Preview page defaults to the `aa_internal` tenant** (`00000000-0000-0000-0000-000000000001`)
  — the only tenant with real tour/atom data today (AA-301 STEP 0, re-confirmed).

## Tradeoffs

- **Frontend eslint is not part of this repo's actual CI gate** (`.github/workflows/ci.yml`'s
  `lint` job only runs Python `flake8` on `api/`/`services/`/`shared/` — verified by reading
  the workflow file). The local `eslint` config flags a `react-hooks/set-state-in-effect` rule
  on several `useEffect` calls in the new pages — but the exact same rule already fires on
  untouched pre-existing files (`admin/review/page.tsx`, `admin/dashboard/page.tsx`,
  `AdminSidebar.tsx`'s own pre-existing code at the line I didn't touch), confirmed via
  `git diff` showing my only change there is the nav-array line. Not a regression, and not
  something CI would catch either way. What DOES matter for a real Vercel deploy —
  `npx tsc --noEmit` and `npx next build` — both pass clean, including the two new routes
  appearing correctly in the build's route list.
- **Second COUNT(*) query per list call**: a real cost (2 DB round-trips instead of 1) paid
  specifically to reuse `Pagination.tsx` as instructed. Acceptable for an internal
  low-traffic curation tool; would need reconsidering if this endpoint ever became
  high-traffic.

## Addendum — real production 500 on `preview-slotgrid`, root cause + fix

Found live via CloudWatch (`/ecs/aa-cis-dev`) after merge: `GET /admin/atoms/preview-slotgrid`
was crashing with a real 500. Full traceback pointed at
`services/acp_planning/quarter.py::_row_to_atom()`:

```
pydantic_core._pydantic_core.ValidationError: 2 validation errors for AtomRecord
cooldown_until
  Input should be a valid dictionary [type=dict_type, input_value='{}', input_type=str]
usage_log
  Input should be a valid list [type=list_type, input_value='[]', input_type=str]
```

**Root cause**: `tour_atoms.cooldown_until`/`.usage_log` are JSONB columns. This app's asyncpg
connections have no jsonb codec registered, so JSONB always arrives as a raw JSON string, not
a parsed dict/list — the exact same gap AA-314 already found and fixed for `src_highlights`
elsewhere (`api/routers/v1_tours.py`). `_row_to_atom()` assumed asyncpg handed back real
Python objects; it didn't.

**Why 87+26 existing tests didn't catch it**: every mock in `test_aa301_quarter.py`/
`test_aa300_admin_atoms.py` built DB-row fixtures with real Python `{}`/`[]` literals instead
of the actual string shape asyncpg produces — a mock-fidelity gap, not a logic gap.

**Fixed**:
- `services/acp_planning/quarter.py`: new `_parse_jsonb()` helper, used in `_row_to_atom()`
  for both fields.
- `api/routers/admin_atoms.py::_safe()`: same root cause, different symptom — `list_atoms()`
  and `patch_atom()` both return `media` (also JSONB) unparsed. Not a crash (JS silently reads
  `undefined` off a string instead of throwing), but `atom.media?.has_photo` on the frontend
  never reflected real data. Fixed in the same pass.
- Grepped both files for every other `row["..."]` access — confirmed no other unparsed JSONB
  reads exist (`runway.py`'s `_row_to_trip()` only touches non-JSONB `v_trip_registry`
  columns).
- Fixed the mock-fidelity gap at its root, not just added an isolated regression test:
  `test_aa300_admin_atoms.py`'s `_atom_row()`/`_atom_db_row()` fixtures now use JSON-string
  JSONB values (matching real asyncpg), and a new `TestParseJsonb`/
  `TestFetchAtomsByTripDbWrapper` in `test_aa301_quarter.py` exercises `_row_to_atom()`/
  `fetch_atoms_by_trip()` directly against the real asyncpg string shape — the layer that
  previously had zero dedicated tests.
- Verified the fix would have been caught: temporarily `git stash`-reverted just the source
  fix (not the tests) and reran — the new tests fail to even collect (`ImportError: cannot
  import name '_parse_jsonb'`), confirming tight coupling between the fix and its tests.
- Verified against the real backend, not just unit tests: copied the fixed
  `services/acp_planning/*.py` into a scratch directory inside the real ECS container
  (AA-305's own established technique) and called `runway_map() → plan_quarter() →
  approve_quarter_plan() → allocate_month()` directly against the real dev DB. Real result:
  `quarter_plan.trip_ids = 5`, `destination_shares = {'South Korea': 0.61, 'Sri Lanka': 0.39}`,
  `slot_grid.slots = 16` with real atom text/keyword_seed from actual South Korea/Sri Lanka
  tour content, and the atom-floor `capacity_note` firing correctly for two thin trips. No 500.
- Full suite: 684 passed (684 = 676 prior + 8 new), 0 failed. `flake8` (CI-exact flags): clean.
