# AA-300 — Atom Curation UI (N3)

## Decisions

- **Router**: new `api/routers/admin_atoms.py`, prefix `/admin`, imports `verify_admin_secret`
  from `admin.py` (does not redefine it) — following the exact established convention
  `admin_pipeline.py` itself documents in its own file header ("Auth: x-admin-secret header
  only (no tenant JWT accepted)"). Registered in `api/main.py` the same way as the other 4
  admin router files (`admin`, `admin_pipeline`, `admin_acp_proxy`, `admin_settings`).
- **Auth**: X-Admin-Secret-only, not the dual X-Admin-Secret-or-JWT pattern `v1_atoms.py` uses.
  The existing frontend BFF proxy (`frontend/app/api/admin/[...path]/route.ts`) only ever sends
  `X-Admin-Secret` — confirmed by reading it — never a JWT, so the dual pattern would be dead
  code here. `POST /v1/atoms/decompose` (the one endpoint using dual auth) has zero callers
  anywhere in the repo (grep-confirmed) — not a load-bearing precedent to imitate for an
  internal CIS-admin tool.
- **Proxy**: reused the existing `/api/admin/[...path]/route.ts` as-is — it is a generic
  catch-all that already forwards any `/api/admin/*` path to `/admin/*` on the backend. No
  second proxy route created.
- **`v1_atoms.py` untouched**: confirmed by a dedicated regression test
  (`TestNoV1AtomsRegression`) — this issue is a purely additive, separate resource.
- **No new `tour_atoms` migration**: `distinctiveness`, `visual_potential`, `starred`,
  `deleted`, `media` all already exist (migration 079) and `starred`/`deleted` are already
  consumed by AA-301's allocator (`services/acp_planning/allocator.py`) — confirmed live in
  STEP 0 recon before writing any code.
- **"unreviewed" filter**: derived from `updated_at == created_at` rather than a new column.
  `tour_atoms` has no `reviewed`/`reviewed_at` field, and both timestamps are set to the same
  `now()` in the same INSERT statement (`v1_atoms.py`, migrations 079/084/085) — Postgres
  resolves multiple `now()` calls in one statement to the same transaction timestamp, so they
  stay exactly equal until the first `PATCH` touches the row (which explicitly sets
  `updated_at = now()`). Self-chosen, not specified by the issue text.
- **"thin tour" filter**: reuses the same `tour_atom_count` computed via a JOIN subquery
  (count of non-deleted, non-empty-marker atoms per `tour_id`), compared against
  `THIN_TRIP_ATOM_MIN` (imported from `services.acp_shared.atom_constants` — the same constant
  N5's B5 fix already uses, not redefined).
- **Soft delete = existing `deleted` column**, not `is_empty_marker` (that marks "this
  decompose run legitimately produced zero atoms," a structurally different fact — confirmed
  by `v1_atoms.py`'s own comment on the distinction). "Star" = existing `starred` column.
  Neither needed a migration.
- **Total-count query added to `GET /admin/atoms`**: the existing `Pagination.tsx` component
  (reused as required) needs a total matching-filter item count, not just the current page's
  row count, to compute page numbers — added a second `COUNT(*)` query using the same `WHERE`
  clause as the paginated `SELECT`. Not specified by the issue text; a direct consequence of
  reusing `Pagination.tsx` as instructed rather than building a new pagination UI.
- **Keyboard shortcuts (X = delete, S = star) bind to the currently-hovered card**, not a
  separately-selected/focused one — no extra click needed to "select" a card first, matching
  the issue's own "mỗi giây đều đáng" (every second counts) framing for a 200-tour curation
  session. Ignored while a text input/textarea is focused (editing an atom's text) so typing
  "x"/"s" doesn't fire a shortcut by accident.
- **Preview endpoint hardcodes `markets=["US"], channels=["blog"],
  capacity_posts_per_week=4`** and auto-approves the quarter plan as `"admin-preview-demo"` —
  the same tenant-config gap AA-301's own implementation notes already flagged (no tenant
  campaign-config table exists yet). Auto-approval here does NOT weaken Gate B for the real
  N5/N6 production path — `approve_quarter_plan()` still must be called explicitly by
  something, and this demo endpoint is read-only relative to the database (no INSERT/UPDATE
  anywhere in `runway_map()`/`plan_quarter()`/`allocate_month()`), so nothing persists from a
  demo "approval."
- **Preview page defaults to the `aa_internal` tenant** (`00000000-0000-0000-0000-000000000001`)
  — the only tenant with real tour/atom data today (AA-301 STEP 0, re-confirmed).

## Tradeoffs

- **Frontend eslint is not part of this repo's actual CI gate** (`.github/workflows/ci.yml`'s
  `lint` job only runs Python `flake8` on `api/`/`services/`/`shared/` — verified by reading
  the workflow file). The local `eslint` config flags a `react-hooks/set-state-in-effect` rule
  on several `useEffect` calls in the new pages — but the exact same rule already fires on
  untouched pre-existing files (`admin/review/page.tsx`, `admin/dashboard/page.tsx`,
  `AdminSidebar.tsx`'s own pre-existing code at the line I didn't touch), confirmed via
  `git diff` showing my only change there is the nav-array line. Not a regression, and not
  something CI would catch either way. What DOES matter for a real Vercel deploy —
  `npx tsc --noEmit` and `npx next build` — both pass clean, including the two new routes
  appearing correctly in the build's route list.
- **Second COUNT(*) query per list call**: a real cost (2 DB round-trips instead of 1) paid
  specifically to reuse `Pagination.tsx` as instructed. Acceptable for an internal
  low-traffic curation tool; would need reconsidering if this endpoint ever became
  high-traffic.

## Addendum — real production 500 on `preview-slotgrid`, root cause + fix

Found live via CloudWatch (`/ecs/aa-cis-dev`) after merge: `GET /admin/atoms/preview-slotgrid`
was crashing with a real 500. Full traceback pointed at
`services/acp_planning/quarter.py::_row_to_atom()`:

```
pydantic_core._pydantic_core.ValidationError: 2 validation errors for AtomRecord
cooldown_until
  Input should be a valid dictionary [type=dict_type, input_value='{}', input_type=str]
usage_log
  Input should be a valid list [type=list_type, input_value='[]', input_type=str]
```

**Root cause**: `tour_atoms.cooldown_until`/`.usage_log` are JSONB columns. This app's asyncpg
connections have no jsonb codec registered, so JSONB always arrives as a raw JSON string, not
a parsed dict/list — the exact same gap AA-314 already found and fixed for `src_highlights`
elsewhere (`api/routers/v1_tours.py`). `_row_to_atom()` assumed asyncpg handed back real
Python objects; it didn't.

**Why 87+26 existing tests didn't catch it**: every mock in `test_aa301_quarter.py`/
`test_aa300_admin_atoms.py` built DB-row fixtures with real Python `{}`/`[]` literals instead
of the actual string shape asyncpg produces — a mock-fidelity gap, not a logic gap.

**Fixed**:
- `services/acp_planning/quarter.py`: new `_parse_jsonb()` helper, used in `_row_to_atom()`
  for both fields.
- `api/routers/admin_atoms.py::_safe()`: same root cause, different symptom — `list_atoms()`
  and `patch_atom()` both return `media` (also JSONB) unparsed. Not a crash (JS silently reads
  `undefined` off a string instead of throwing), but `atom.media?.has_photo` on the frontend
  never reflected real data. Fixed in the same pass.
- Grepped both files for every other `row["..."]` access — confirmed no other unparsed JSONB
  reads exist (`runway.py`'s `_row_to_trip()` only touches non-JSONB `v_trip_registry`
  columns).
- Fixed the mock-fidelity gap at its root, not just added an isolated regression test:
  `test_aa300_admin_atoms.py`'s `_atom_row()`/`_atom_db_row()` fixtures now use JSON-string
  JSONB values (matching real asyncpg), and a new `TestParseJsonb`/
  `TestFetchAtomsByTripDbWrapper` in `test_aa301_quarter.py` exercises `_row_to_atom()`/
  `fetch_atoms_by_trip()` directly against the real asyncpg string shape — the layer that
  previously had zero dedicated tests.
- Verified the fix would have been caught: temporarily `git stash`-reverted just the source
  fix (not the tests) and reran — the new tests fail to even collect (`ImportError: cannot
  import name '_parse_jsonb'`), confirming tight coupling between the fix and its tests.
- Verified against the real backend, not just unit tests: copied the fixed
  `services/acp_planning/*.py` into a scratch directory inside the real ECS container
  (AA-305's own established technique) and called `runway_map() → plan_quarter() →
  approve_quarter_plan() → allocate_month()` directly against the real dev DB. Real result:
  `quarter_plan.trip_ids = 5`, `destination_shares = {'South Korea': 0.61, 'Sri Lanka': 0.39}`,
  `slot_grid.slots = 16` with real atom text/keyword_seed from actual South Korea/Sri Lanka
  tour content, and the atom-floor `capacity_note` firing correctly for two thin trips. No 500.
- Full suite: 684 passed (684 = 676 prior + 8 new), 0 failed. `flake8` (CI-exact flags): clean.

## Addendum 2 — redesign backend: `/atoms/summary` + bulk PATCH (Nghiep's direct UI feedback)

Backend half of the redesign approved after Nghiep reviewed the live curation UI: table rows +
accordion-by-tour + multi-select + full dashboard + load-more instead of Pagination.tsx.
Frontend half not started yet (strict-synchronous: stop after backend, wait for signal).

- **Branch stacked on `aa-300-fix-jsonb-parse-500` (PR #86), not `main`** — this new work
  touches the same file (`admin_atoms.py`) as the still-unmerged JSONB fix; branching from
  main would mean redoing/duplicating that fix or hitting avoidable merge conflicts. This
  PR's diff will include #86's commit until #86 merges, then shrink to just its own commits
  — deliberate, not an accident.
- **`GET /admin/atoms/summary` as a separate endpoint, not fields bolted onto `GET /atoms`** —
  the dashboard (total/breakdown/reviewed/by-tour) only needs to change when underlying data
  changes (a star/delete/bulk action), not on every filter click or load-more page fetch.
  Folding it into the paginated list would mean recomputing 3 extra aggregate queries on
  every single filter/pagination round-trip for no reason. Fewer total queries over a real
  editing session with the split, even though it's technically "one more endpoint" —
  explicit tradeoff, matches what the issue asked me to decide with justification.
- **`PATCH /admin/atoms/bulk` registered BEFORE `PATCH /admin/atoms/{atom_id}`** — this
  repo's own CRITICAL rule (CLAUDE.md: "/{id}/full MUST come BEFORE /{id}") applies here:
  FastAPI would otherwise greedily match `"bulk"` as `{atom_id}` on a `PATCH /atoms/bulk`
  request. Verified via `app.routes` inspection (not just visual code order) — a dedicated
  regression test (`test_bulk_route_registered_before_dynamic_atom_id_route`) locks this in.
  One UPDATE ... WHERE atom_id = ANY($1), not N sequential calls — the whole point of "bulk."
- **Bulk endpoint only supports `starred`/`deleted`, not `text`** — editing many atoms' text
  to the identical value isn't a real use case; self-chosen, not specified by the issue.
- **`reviewed_count`** uses the same `updated_at != created_at` predicate already established
  in `list_atoms()`'s `unreviewed_only` filter — one definition of "reviewed," not a second
  one invented for the dashboard.
- Verified the new SQL against the real dev DB directly (not just mocks), given the PR #86
  lesson about mock-vs-real asyncpg shape: `BREAKDOWN: [{'LOW': 235}]` (matches the known
  100%-LOW-today state, AA-317 not built), `TOTALS: {total: 235, reviewed: 1}`,
  `BY_TOUR: 25 tours, 8 thin (< 5 atoms)`, and a 0-match dry-run bulk UPDATE to confirm the
  `ANY($1)` SQL is syntactically valid without touching real rows.
- Full suite: 693 passed (693 = 684 prior + 9 new), 0 failed. `flake8` (CI-exact flags): clean.

## Addendum 3 — CI failure on `test_bulk_route_registered_before_dynamic_atom_id_route`

CI (run on this branch, commit `a5d9649`) failed with:
`ValueError: '/admin/atoms/bulk' is not in list` — same test, passed locally every time it
was run. Investigated rather than assumed a merge/rebase mistake.

**Root cause, confirmed by reproduction, not guessed**: `requirements.txt` pins
`fastapi>=0.110.1` — an open floor, not an exact version. This dev environment's `.venv` has
been frozen at `fastapi==0.110.1`/`starlette==0.37.2` for a long time; CI does a completely
fresh `pip install -r requirements.txt` on every run, so it resolves whatever is newest at
that moment — `fastapi==0.139.2`/`starlette==1.3.1` right now, a major-version jump on
Starlette (0.x → 1.x). Reproduced exactly by building a clean venv from this repo's own
`requirements.txt` + `requirements-acpcore.txt` (the same two install commands CI's own
workflow runs) and re-running the failing test in it — same `ValueError`, confirmed before
writing any fix.

Root cause of the `ValueError` itself: newer FastAPI's `include_router()` wraps included
routers in an opaque `_IncludedRouter` object instead of flattening each `Route` into
`app.routes` at include-time. Printing `app.routes` in the CI-matching venv showed most
routes replaced by these wrapper objects (no `.path`/`.methods` attributes) — the previous
version of this test walked `app.routes` directly, which is an internal implementation detail
that changed between these two Starlette major versions. **The actual route registration
order in `admin_atoms.py` was never wrong** — confirmed by writing a version-independent
replacement test and running it successfully in both the old (0.110.1) and new (0.139.2)
environments side by side.

**Fixed**: rewrote the test to make a real HTTP request through a minimal, lifespan-free
`FastAPI()` instance mounting just `admin_atoms.router` (via `fastapi.testclient.TestClient`),
and assert on the real response shape (`{"updated_count": ...}` only the bulk handler
produces) rather than introspecting `app.routes`. Deliberately did NOT use the real
`api.main.app` + `TestClient`'s lifespan — `api/main.py`'s `lifespan()` dials real
Redis/Postgres on startup, which would hang/fail in the actual Unit Tests CI job (no
Redis/Postgres service available there, confirmed by reading `ci.yml`'s job definition).
Verified the fix passes in both a plain local run and a full run of `tests/unit/` inside the
CI-matching venv (693 passed, 0 failed, in both) — not merely re-running the one test.

**Not fixed here, flagged as a separate, real, pre-existing risk**: `fastapi>=0.110.1` being
an open floor means CI (and this project's Docker image build, which also runs
`pip install -r requirements.txt` fresh) has never actually been pinned to the same FastAPI
version this dev environment runs against — this is the first test to notice because it's the
first one to touch FastAPI/Starlette internals directly, but the underlying "unpinned floor
dependency, fresh install every CI run" pattern applies to every dependency in
`requirements.txt` equally and predates this PR. Not changed here — pinning is a deliberate
dependency-management decision affecting the whole app, not a narrow test fix.

## Addendum 4 — 4 small UI polish fixes from Nghiep's direct browser feedback on the redesign

Frontend-only, no backend/migration changes. N4-N6 UI Phase 2 (input selection for N5, runway
map as a year-table, parameter config, atom detail-in-slot — larger gaps flagged in the same
review) explicitly split out to AA-323 (ACPv2 Phase 2 project), not touched here.

- **Sort dropdown**: added a second `FilterBar` filter (`Sắp xếp`) next to Distinctiveness,
  sorting `orderedSections` (tour-level, not atom-level — "số atom"/"% chưa review" are
  per-tour stats from `summary.by_tour`) by 4 keys. Default (`sortBy === ""`) returns the
  existing `summary.by_tour` order untouched — explicit per the issue's "không đổi behavior
  mặc định." `FilterBar.tsx` gained an optional `allLabel` prop (defaults to `"All"`, so its
  one other consumer — the Distinctiveness filter on this same page — is unaffected) because
  its hardcoded `"{label}: All"` default option read wrong for a sort field; used `"Mặc định"`
  here. Grepped: `master-content/page.tsx` imports `FilterBar` but never renders it (dead
  import, pre-existing, not touched) — this page is the component's only real consumer, so
  extending its prop shape has zero blast radius elsewhere.
- **StatCard grid**: `display:flex` (unequal card widths, content-sized) → `display:grid,
  gridTemplateColumns: repeat(5,1fr)`, matching `master-content/page.tsx`'s own
  `repeat(4,1fr)` StatCard row exactly (same pattern, different column count — curation has 5
  cards, master-content has 4). Grid's default `stretch` alignment fills each column evenly
  even through the extra opacity-wrapper `<div>` around HIGH/MED/LOW. Shortened the
  zero-count sub-text `"none yet (AA-317 pending)"` → `"chưa có (AA-317)"`.
- **Page max-width**: grepped every other admin `<main>` (`dashboard`, `master-content`,
  `tenants`, `upload`, `review`, `s1-rewrite`) — none cap width at all (`flex:1`, padding only,

  `maxWidth` but no centering, so on a wide monitor the capped content sat flush left against
  the sidebar with a large dead gutter on the right — added `margin: "0 auto"`, which
  flexbox resolves correctly on a `flex:1` item (auto margins absorb the leftover free space
  post-flex-and-clamp, the standard flex-centering technique) to center it instead. Scoped to
  `page.tsx` only, per the issue text ("trang curation") — did not touch `preview/page.tsx`,
  which wasn't in scope and may want the same fix as a separate, explicit follow-up.
- **Floating bar covering Load More**: chose the padding-bottom option over moving the button,
  per the issue's own "ít xáo trộn code nhất" instruction — one conditional style property
  (`paddingBottom: selectedIds.size > 0 ? FLOATING_BAR_CLEARANCE : 0`) on the existing scroll
  container, vs. restructuring the section-map/Load-More/floating-bar layout. `92`px
  clearance sized from the floating bar's own actual rendered dimensions (`padding: "10px
  16px"` + one line of 13px/600-weight text + `bottom: 24`), not guessed.
