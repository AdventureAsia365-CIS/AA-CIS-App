# AA-210 — Guard `pipeline_runs` accounting UPDATE against non-UUID `batch_id`

## Problem
`_execute_run_tour` and `_run_tour_safe` in `api/routers/admin_pipeline.py` run two
`UPDATE shared.pipeline_runs ... WHERE batch_id = $N::uuid` statements:

- accounting UPDATE — `admin_pipeline.py:484` (cost/tokens/model rollup, in `_execute_run_tour`)
- failure-path UPDATE — `admin_pipeline.py:531` (mark batch `failed`, in `_run_tour_safe`)

`TourRunRequest.batch_id` is typed `str` (`admin_pipeline.py:88`) and accepts any value.
When a caller passes a non-UUID label (e.g. verification runs using `"verify-s68-sonnet"`),
the `$N::uuid` cast raises `asyncpg.exceptions.DataError: invalid input ... invalid UUID`.

The crash fires **after** content generation, judge, and gold export have already
committed (asyncpg autocommits per statement), so:
- content/publish are unaffected, but
- the `pipeline_runs` accounting never records, and
- an uncaught traceback is logged; on the synchronous `/admin/run-tour` path it would
  surface as a 500 (masked in practice by the API Gateway 29s timeout).

Reproduced 4× during AA-193 brand verification runs (Atlas/Terra/Trail Pulse/WildKind),
all using `batch_id="verify-s68-sonnet"`.

## Root cause
Confirmed via `information_schema`: `shared.pipeline_runs.batch_id` is genuinely
**`uuid`** type. So the `::uuid` cast is *correct* for the column. The defect is on the
input side — missing validation that `req.batch_id` is a UUID before it reaches the cast.

## Fix
Runtime UUID guard, no schema/cast change:
- Add module-level `_is_uuid(value) -> bool` helper.
- Accounting UPDATE: require `_is_uuid(req.batch_id)` in the existing
  `if cost_usd > 0 or tokens_in > 0:` condition; `else` log
  `pipeline_runs_accounting_skipped` (batch_id + reason) at info level.
- Failure-path UPDATE: only run when `_is_uuid(tour_req.batch_id)`; otherwise log a skip.

## Decisions
- **Reuse existing `from uuid import UUID`** instead of adding `import uuid as _uuid`
  (would be a redundant duplicate import). Helper calls `UUID(str(value))`.
- **No semicolon one-liner** in the helper — `UUID(...); return True` trips flake8 `E702`.
  Written as a normal multi-line try/except (CI requires flake8 clean).
- Guard, not a Pydantic `batch_id: UUID` type-tighten: tightening would 422-reject the
  string verification batch_ids the team uses ad-hoc. Runtime skip is friendlier and
  still safe.

## Changed (vs. original request)
- Helper uses the already-imported `UUID` (not a new `import uuid as _uuid`).
- Helper body is multi-line (flake8 `E702`), not the `;`-separated form.

## Tradeoffs
- For non-UUID batch_ids the accounting UPDATE is silently skipped (logged). Acceptable:
  ad-hoc single-tour runs have **no matching `pipeline_runs` row anyway** (those rows are
  minted by ingestion), so the UPDATE would match 0 rows even with a valid UUID.
- Real production ingestion batch_ids are UUIDs → unaffected; behavior unchanged there.

## Should know (reviewer)
- This does not touch the export/approval gate or `review_queue` (separate, larger gaps
  found in the same investigation — to be filed independently).
- Both UPDATEs guarded identically so behavior is consistent across success and failure paths.

## Verify
1. `flake8 api/routers/admin_pipeline.py --max-line-length=120` → clean.
2. Unit: `_is_uuid` truth table — valid uuid str → True; `"verify-s68-sonnet"` → False;
   `None` → False.
3. Behavioral: `_execute_run_tour` with a non-UUID batch_id completes without raising and
   logs `pipeline_runs_accounting_skipped` (mock conn.execute asserted not awaited for the
   pipeline_runs UPDATE).
4. Regression: with a valid UUID batch_id the UPDATE still executes (mock asserted called).
