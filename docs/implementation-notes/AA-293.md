# AA-293 (Bug A) — asyncpg JSONB codec gap in `_revalidate_tour`

## Decisions
- Chose manual per-field `json.loads()` (option b in the task) over a connection-level
  `set_type_codec('jsonb', ...)` (option a). `_revalidate_tour` binds `json.dumps(...)`
  Python strings to `::jsonb` columns in its own `INSERT INTO quality_scores` a few
  lines below the reads — a codec registered on the same connection would call its
  encoder (`json.dumps`) on those already-dumped strings a second time, corrupting the
  write. Manual decode avoids that side effect entirely and matches the codebase's
  existing convention: `admin_settings.py` already has an identical local `_jsonb()`
  helper for the same reason, and `seo_intelligence/handler.py` does the same
  `json.loads(row["field"] or "[]")` pattern by hand. Grepped the whole repo for
  `set_type_codec` — zero hits anywhere, including the main app pool
  (`api/main.py`, `shared/repository/database.py`) — so this is a systemic gap, not
  something local to `_revalidate_tour`; today nothing anywhere registers one.
- `aa_itineraries` is `TEXT`, not `JSONB` (confirmed in `api/migrations/002/003_schema_v3.sql`)
  — deliberately left out of the decode list. Running `json.loads()` on it would
  break every itinerary that isn't valid JSON (i.e. all of them — it's day-by-day
  prose text).

## Changed
- `api/routers/admin_pipeline.py:742` (`_revalidate_tour`): added local `_jsonb()`
  helper; applied to all jsonb columns the function reads —
  `aa_highlights`, `seo_keywords_used`, `og_tags`, `metadata` (generated_content) and
  `keyword_ideas`, `top_keywords` (seo_context). Replaced the ad-hoc
  metadata-only `isinstance(str)` guard (which already existed, proving the author
  knew about the gap for that one field but didn't generalize it) with the shared
  helper for consistency.

## Tradeoffs
- Did not add a codec to the main app pool either, even though it has the same gap —
  out of scope for this bug (AA-293 Bug A is scoped to `_revalidate_tour`
  specifically, discovered via AA-251 batch verification). If the main pool's
  reads elsewhere are ever found to have the same silent-string bug, that's a
  separate ticket — fixing it broadly risks the same double-encode hazard
  everywhere `json.dumps(...)::jsonb` params are already in use.

## Should know
- This bug was discovered while batch-verifying the AA-251 fix via `/revalidate` on
  22 tours: `HIGHLIGHTS_NOT_LIST` false-fired on 100% of them (a code that never
  appeared in any of their prior failure_codes), and `DFS_INTENT_UNDERUSED` was
  masked/corrupted for the one tour that had a `seo_context` row — the `for kw in
  seo_kws` loop was iterating the raw JSON string character-by-character instead of
  its actual keywords.
- A second, independent issue was also found during that same verification: **21 of
  22 tested tours have no `seo_context` row at all for their current `tour_id`** —
  the DFS_INTENT_UNDERUSED check is silently *skipped* for them (`seo_kws` falsy),
  not passed. That is NOT fixed by this ticket — it's a data/backfill question, not
  a decoding bug — and should be re-verified against `/revalidate` again now that
  this codec bug is fixed, or verified via a fresh `/run-tour` instead.
