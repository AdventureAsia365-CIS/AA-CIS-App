# AA-242 — Regenerate wiring (review queue → run-tour-async → supersede)

Wire the review-queue "Regenerate" action end to end: rerun the full pipeline for a tour,
then close the old review row as `superseded` when the regenerated version comes back
publishable.

## Root cause / why the wiring was missing

- `TourRunRequest` needs 3 required fields — `tour_id`, `batch_id`, `tenant_id` — plus
  `model_tier` to avoid a silent haiku→sonnet upgrade (the AA-237 opt-in gate,
  `allow_auto_upgrade`, defaults OFF).
- `GET /admin/review-queue` did not surface `requested_tier` or a batch id, so the FE had
  no way to construct a correct `run-tour-async` request. The Regenerate button was a static
  placeholder ("tracked in AA-242").

## Decisions

- **Mint a NEW `batch_id` every regenerate** (`crypto.randomUUID()`), never reuse the
  original. Reusing it risks a `find_active_duplicate` dedup false-positive that would drop
  the rerun.
- **`allow_auto_upgrade` always `false`** — regenerate must run the tier the reviewer sees,
  no silent sonnet re-run.
- **Publishable verdict = real backend gate.** FE keys the supersede decision on
  `gc.status === "approved"` (written by `_is_publishable`), never on a client-side score
  heuristic. Added `gc.status` to the queue SELECT specifically for this.
- **Migration 073 adds `'superseded'` to `review_status_enum`**, distinct from `'rejected'`
  (reviewer judged it bad) and `'skipped'` (dead value — old timeout/auto-skip, unused).
  `superseded` = the row was replaced by a newer regenerated version, not judged.

## Changed

- **Backend `api/routers/admin_pipeline.py`:**
  - `admin_review_queue` SELECT + response dict now surface `requested_tier`,
    `raw_tours_batch_id` (`rt.batch_id`), and `status` (`gc.status`).
  - New `POST /review-queue/{review_id}/supersede` — `UPDATE ... SET review_status =
    'superseded' WHERE id = $1::uuid AND review_status = 'pending'`; `UPDATE 0` → 409.
    Uses the `request.app.state.pool` pattern (matches the adjacent GET route), not a raw
    `asyncpg.connect`.
- **Frontend `frontend/app/admin/review/page.tsx`:**
  - `RegenerateModal` is now a real confirm modal: warns it reruns the full pipeline on real
    models (actual Bedrock spend), encourages fixing by hand (Edit) first, and states the
    current review row will auto-close as `superseded` if the rerun is publishable. When
    `requested_tier` is null it forces an explicit haiku/sonnet pick — no silent default.
  - `doRegenerate`: `POST /api/admin/run-tour-async` → 202 → poll via the shared `pollJob`
    (reused, not reimplemented) → on success refetch the queue, find the newest row for the
    same `tour_id`, and if its `status === "approved"` call `supersede` on the OLD review id.
    Any non-success job outcome surfaces an error and does NOT supersede/refetch.
- **Migration `api/migrations/073_review_status_superseded.sql`** — `ALTER TYPE
  review_status_enum ADD VALUE IF NOT EXISTS 'superseded'`. Already applied to the DB.
- **Test `tests/unit/test_aa242_regenerate_supersede.py`** — 3 cases.

## Tradeoffs

- `doRegenerate` lives co-located inside `RegenerateModal` (like `revalidate()` lives inside
  `ReviewEditor`), because it needs the modal-local tier/running/msg state. It takes an
  `onReload` prop (= the page's `load`) to refresh the queue after completion.
- Regenerate does NOT overwrite a version — the pipeline always INSERTs a new `version_num`
  (verified in STEP 0). The old review row is only closed (`superseded`) when the new version
  reaches `status='approved'`. If it lands in `hitl` instead, both versions stay pending so
  the reviewer can compare and decide.

## Should know

- **`raw_tours.batch_id` ≠ `metadata.batch_id`.** `raw_tours.batch_id` is the real ingestion
  batch (a genuine UUID); `metadata.batch_id` can be an ad-hoc non-UUID label from historical
  manual runs. We surface `raw_tours.batch_id` as display context only — it is NOT reused for
  the regenerate request (we mint a fresh one). Do not conflate them.
- **The supersede endpoint has NO UUID guard** — it hands `review_id` straight to `$1::uuid`.
  A malformed id makes asyncpg raise `InvalidTextRepresentationError` (⊂ `DataError`) on the
  cast; the endpoint does not catch it, so it propagates (surfaces as 500 upstream, NOT
  409/422). `test_supersede_invalid_uuid` documents this as intended behavior — adding a guard
  is out of AA-242 scope.
