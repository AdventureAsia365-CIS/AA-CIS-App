# AA-289 (prompt versioning + eval gate) + AA-288 (prompt caching) — S1-old and S1-from-atom

## STEP 0 findings (2026-07-23)

Both issues' original text assumed things that turned out to be stale once checked against
real code:

1. **S1-old already has prompt caching wired.** `shared/llm_client/client.py::_call_bedrock`
   is called with `use_cache=True` for both Sonnet/Haiku tiers, and `build_cached_system_prompt`
   already sets `cache_control: {type: ephemeral}`. But `_call_bedrock` parsed
   `cache_read_input_tokens`/`cache_creation_input_tokens` from the Bedrock stream and then
   discarded them — `LLMResponse` had no field for them, so they never reached the DB. The L2
   few-shot cache path (`build_cached_messages`) is dead code for a different reason: the real
   production call site (`admin_pipeline.py:837`) always passes `"few_shots": []` — there is no
   few-shot data flowing through the pipeline today, so wiring L2 caching would cache nothing.
   Left alone, documented here rather than silently expanded into scope.
2. **The "29 lessons" (AA-131) load live per-run, not baked into the static prompt file.**
   Found in `admin_pipeline.py::_execute_run_tour`: queried fresh from
   `shared.pipeline_lessons` and appended into `brand_rules["system_prompt"]` before the graph
   runs — i.e. already part of the same cached system-prompt block. No separate caching work
   needed.
3. **`pipeline_runs` is the wrong table for this.** It's a batch/ingestion aggregate
   (`services/ingestion/handler.py` / `services/export/handler.py`), not per-tour — no
   `quality_score` to join against. The real per-tour write path is
   `silver_aa_internal.generated_content.metadata` (JSONB, via `_build_generated_metadata()`),
   which already holds `quality_score`. **Decision (Nghiep, 23/07/2026): prompt_version +
   cache_read_tokens/cache_write_tokens go on `generated_content.metadata`, not
   `pipeline_runs`** — no migration needed there, and no duplicate write path to a second table
   that could drift out of sync.
4. **S1-from-atom had zero persistence before this PR** (stateless by AA-306 design). New
   minimal table: `acp_contract.s1_from_atom_runs` (migration 088). No `quality_score` column —
   this pipeline has no judge/quality-scoring node of its own yet, a real gap left visible
   rather than faked with a placeholder value.
5. **Palmyra X5 does not support prompt caching — confirmed via official AWS docs, not
   assumption.** Fetched the exact Bedrock request/response schema for
   `writer.palmyra-x5-v1:0`: the full request parameter table (`messages, temperature, top_p,
   max_tokens, min_tokens, stop, seed, presence_penalty, frequency_penalty`) has no caching
   field, and the response `usage` object has no `cached_tokens`/`cache_write_tokens` (unlike
   Bedrock's GPT-5.6 models, which do). Also absent from Bedrock's own prompt-caching
   supported-models table. **AA-288 closed for S1-from-atom — versioning only.**
6. **Bedrock Claude Sonnet 4.5/Haiku 4.5 are cache-eligible per official docs**, minimum
   checkpoint 4,096 tokens (not 1,024). See "Decisive live finding" below for why this bar is
   moot in current production reality.
7. **Golden Tours 20 fixture was a real access gap.** `data/CIS_Golden_Tours_20_v1.xlsx` was
   only an untracked local file (`data/` gitignored) — unreachable from a GitHub Actions
   runner. Uploaded to `s3://aa-cis-bronze-005097885195/fixtures/CIS_Golden_Tours_20_v1.xlsx`.
   Its 20 rows are shaped for old-S1's raw-itinerary input (no `atom_id` data at all) — cannot
   evaluate S1-from-atom. That pipeline's eval uses the 4 real curated tours already verified in
   AA-306 instead.
8. **Dev RDS is VPC-private** (`PubliclyAccessible: False`, confirmed via
   `aws rds describe-db-instances`) — a GitHub-hosted runner cannot reach it directly. This
   repo's existing CI never touches it either (Integration Tests spin up a local Postgres
   container). The eval-regression workflow is `workflow_dispatch`-only and runs the actual
   eval **inside** the live ECS task via `aws ecs execute-command` — GitHub Actions is only the
   trigger/orchestrator, using the same OIDC role (`aa-cis-dev-role`, confirmed
   `AmazonECS_FullAccess` covers `ecs:ExecuteCommand`) already used by `deploy-dev.yml`.

## Decisive live finding (changes the real conclusion for AA-288/S1-old)

Ran a real tour twice through `_execute_run_tour` to measure caching. Both calls resolved as
`model_used: "satellite-haiku-4-5"` — i.e. **every real call fell through to the acc1 satellite
path (`_call_bedrock_satellite`/`invoke_claude()`), not the direct acc2 Bedrock path
(`_call_bedrock`) where `cache_control` actually lives.** This matches ADR-2026-027: acc2
(`005097885195`) blocks Anthropic models entirely (channel-program policy) — direct Bedrock
calls to `us.anthropic.claude-haiku-4-5-*` on acc2 always fail validation and fall back to the
acc1 satellite, confirmed live: `t2_failed_trying_t3` → satellite retry succeeded.
`bedrock_satellite.py::invoke_claude()` has **no `cache_control` field anywhere** in its request
body.

**Conclusion: S1-old's prompt-caching code is real, correctly wired, and completely unreachable
in production today.** Verified via direct DB query on `generated_content.metadata` for both
calls: `cache_read_tokens: 0, cache_write_tokens: 0` (real zeros, not nulls — the plumbing ran
and correctly recorded that no caching happened). `prompt_version` was correctly identical
(`e09be3b3`) across both calls on the same tour/brand, confirming the hash is deterministic and
the DB write path works end-to-end.

Whether the system prompt alone clears Bedrock's 4,096-token cache-checkpoint minimum is now a
moot question for current production impact — caching cannot activate on the satellite path
regardless of token count. Real measured numbers for context: unbranded system prompt (base
`SYSTEM_PROMPT` + live-loaded lessons) = 3,927 chars; real total per-call input tokens (system +
user combined, from the live Bedrock response `usage.input_tokens`, not a char/4 estimate) =
2,638 and 2,917 across the two calls. The system-prompt-only portion could not be cleanly
isolated from a live call without further instrumentation (Bedrock's `usage` object doesn't
break down system vs. user tokens outside of cache_read/cache_write, which are zero here) — it
is necessarily less than the 2,638–2,917 total.

**Follow-up (not in this PR's scope):** activating caching for S1-old's real traffic would
require adding `cache_control` support to `bedrock_satellite.py::invoke_claude()` itself — a
real code change to the acc1 satellite client, not "surface an already-parsed metric." Flagged
for separate consideration, not silently attempted here.

## Real throttling finding + fix (Palmyra, acc2)

`services/eval/regression.py`'s S1-from-atom eval loop hit `ThrottlingException` on
back-to-back Palmyra calls — reproduced live twice, once with zero spacing between tours, once
with a fixed 25s gap (still not reliable). Root-caused and fixed at the source
(`services/content_generation/s1_from_atom.py::_call_palmyra`): retry-with-backoff
specifically on `ThrottlingException` (3 retries, 30s/60s/90s backoff), not just a longer fixed
sleep in the eval script. This fixes the real `/v1/s1-from-atom/generate` endpoint too, not
just the eval job. Verified live: the full 4-tour eval run hit the throttle on 3 of 4 calls,
recovered via a single 30s-backoff retry each time, and completed successfully end-to-end.

## Should know

- `words_per_citation` in `acp_contract.s1_from_atom_runs` stores as a long-decimal NUMERIC
  from a Python float (e.g. `17.699999999999999...`) — cosmetic float-precision artifact of
  asyncpg's NUMERIC binding, doesn't affect correctness or comparisons. Not rounded at the
  INSERT site; minor nice-to-have, not fixed here.
- `eval-regression.yml` could not be triggered via `gh workflow run` before this PR merges —
  confirmed via `gh workflow list` (doesn't appear) that GitHub only registers
  `workflow_dispatch` workflows once the file exists on the default branch, even when pushed to
  a feature branch. This is a GitHub platform constraint, not a defect in the YAML — the YAML
  follows the exact same OIDC/ECS pattern as the already-working `deploy-dev.yml`. The
  substantive part (the actual eval script, running for real inside ECS, writing real DB rows)
  was fully verified directly; only the thin GitHub Actions wrapper couldn't be live-tested
  pre-merge.
